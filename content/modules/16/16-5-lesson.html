It learns nothing else - not the birthdate, not the Aadhaar number, not even the exact age. The customer's privacy is preserved; the bar's legal obligation is satisfied. 03   Background & Mathematics

From Dates to Arithmetic Circuits

Age verification is a classic range proof problem. You need to prove that a hidden number birthDate (expressed as days since a reference point, e.g., Unix epoch) satisfies the inequality:

todayDate − birthDate ≥ 6570 days  (≈ 18 years × 365 days, simplified)

In a ZK circuit, we cannot directly use inequality (<, >) because they are not natively supported in R1CS (Rank-1 Constraint System). Instead, we perform the subtraction and then prove that the result is less than some maximum bound and that no underflow occurred. The standard approach is to use a comparator circuit that decomposes the numbers into bits and checks the subtraction using a LessThan template from circomlib.

💡 Why 6570 days?

We can choose any reference point, but using a fixed threshold date simplifies the circuit. For example, if today is 01-Jan-2026, the threshold for being 18 is 01-Jan-2008. Instead of computing the difference between two dynamic dates, we pre-compute the threshold date and prove that birthDate ≤ thresholdDate. This is a simpler inequality to express in a circuit. However, for generality, we'll build a circuit that takes today and birthDate as inputs and computes the difference, then checks it's ≥ 6570. Both approaches are valid; the fixed-threshold method is more gas-efficient because the threshold can be a hardcoded constant.

3.1 The LessThan Template

Circomlib provides a LessThan(n) template that takes two n-bit numbers and outputs 1 if the first is strictly less than the second. Internally, it uses a long subtraction with bit decomposition and a check that the final borrow is 0. You will use this template. For age verification, we need to check that today - birthDate >= 6570. We can compute diff = today - birthDate and then check that 6570 - 1 < diff (i.e., diff >= 6570). To avoid negative numbers, we first assert that today >= birthDate using another LessThan call. This is a crucial step: if birthDate is in the future, the proof must fail.

3.2 Preventing Date Manipulation

The circuit's public output is a single bit isAdult. The verifier (the bar's tablet) must be convinced that the proof was generated with the correct today. If the prover can choose an arbitrary today, they could set it to a future date and appear older. The solution is to bind today to a trusted timestamp. In practice, the verifier provides the current date as a public input to the circuit, or the circuit uses the blockchain's block.timestamp (with caveats about miner manipulation). In our exercise, we'll make today a public input that the verifier checks against a recent block timestamp or a trusted server.

3.3 Understanding R1CS (Rank-1 Constraint Systems)

Before we write the Circom circuit, it helps to understand what happens beneath the Circom compiler. When you compile a Circom file, the compiler translates your high-level code into an R1CS - a Rank-1 Constraint System. This is the mathematical representation that the proving system (Groth16, PLONK, etc.) actually works with. Think of R1CS as the "machine code" of zero-knowledge proofs: you write in Circom (a high-level language), and the compiler produces R1CS (the low-level instruction set).

🧩 Analogy - R1CS as a Sudoku Puzzle

Imagine you want to prove to a friend that you solved a Sudoku puzzle without showing the solution. R1CS is like converting every Sudoku rule ("each row has 1-9 without repetition", "each column has 1-9 without repetition", etc.) into individual mathematical equations. Each equation constrains one small part of the puzzle. If you can satisfy all the equations simultaneously, you must have a valid solution. The prover demonstrates that they know values (the "witness") that satisfy every constraint, without revealing those values.

An R1CS constraint has the form:

A * B = C

Here, A, B, and C are linear combinations of the circuit's variables (signals in Circom). Every computation in your circuit - addition, multiplication, comparison - gets broken down into constraints of this form. The key insight is that multiplication requires its own constraint, while addition is "free" (it can be folded into the linear combination). This is why ZK circuit designers count "multiplication gates" or "constraints" as a measure of circuit complexity.

3.4 Worked Example - From Arithmetic to R1CS

Let us walk through a simplified example. Suppose we want to prove that we know a secret number x such that x * x - x = 42 (i.e., x^2 - x = 42). Here is how this single equation becomes R1CS constraints:

Step 1
Flatten the expression into intermediate variables
We introduce an intermediate variable v1 = x * x. Now our computation has two parts: (a) v1 = x * x, and (b) v1 - x = 42. Each multiplication becomes one constraint.
Step 2
Express each step as A * B = C
Constraint 1: (x) * (x) = (v1). This captures the squaring operation. Constraint 2: (v1 - x) * (1) = (42). Multiplying by 1 is a standard trick to express a linear check as an R1CS constraint. Now we have two constraints that together enforce x^2 - x = 42.
Step 3
Write the witness vector
The witness is the complete set of values: [1, x, v1] where the leading 1 is a constant always included. If x = 7, then v1 = 49, and the witness is [1, 7, 49]. The prover knows this witness; the verifier does not. Both constraints evaluate correctly with these values: 7 * 7 = 49 (check) and 49 - 7 = 42 (check).
💡 Why This Matters for Age Verification

In our age verification circuit, the Circom compiler performs this same flattening process automatically. The LessThan template alone generates dozens of R1CS constraints because it decomposes numbers into individual bits and checks each bit position. A typical age verification circuit with 32-bit comparisons produces roughly 200-300 R1CS constraints. The more constraints, the larger the proving key and the longer proof generation takes - but verification time remains nearly constant (~1 ms for Groth16), which is why this scheme is practical for on-chain verification.

04   Technical Deep Dive

The AgeVerification Circuit in Circom

Let's write the circuit step-by-step. We'll use a reference point of days since 01-Jan-1970 (Unix epoch). The inputs will be:

  • Private inputs: birthDay (the user's birthdate as days since epoch).
  • Public inputs: today (the current date in days since epoch), threshold (the minimum required age in days, e.g., 6570).
  • Output: isAdult (1 if age ≥ threshold days, else 0).

The circuit must enforce:

  1. today >= birthDay (the user cannot be born in the future).
  2. diff = today - birthDay.
  3. isAdult = (diff >= threshold) ? 1 : 0.
Circom Circuit for Age Verification
pragma circom 2.0.0;

include "circomlib/circuits/comparators.circom";

template AgeVerification() {
    // Private input: user's birthdate (days since epoch)
    signal input birthDay;

    // Public inputs: current date and age threshold in days
    signal input today;
    signal input threshold;

    // Output: 1 if age >= threshold days, else 0
    signal output isAdult;

    // 1. Ensure today >= birthDay
    // Using LessThan to check that birthDay < today (strictly) OR equal.
    // LessThan(32) checks if birthDay < today; if so, out=1.
    component birthBeforeToday = LessThan(32);
    birthBeforeToday.in[0] <== birthDay;
    birthBeforeToday.in[1] <== today;
    // If birthDay == today, LessThan returns 0. We need to allow equality.
    // So we add an equality check: birthDay == today → also valid.
    component isEqual = IsEqual();
    isEqual.in[0] <== birthDay;
    isEqual.in[1] <== today;
    // Valid if birthBeforeToday.out == 1 OR isEqual.out == 1
    signal validBirth <== birthBeforeToday.out + isEqual.out;
    validBirth === 1;  // must hold

    // 2. Compute difference
    signal diff <== today - birthDay;

    // 3. Check diff >= threshold
    // To prove diff >= threshold, we check that threshold - 1 < diff.
    // That is, (threshold - 1) is strictly less than diff.
    // We need to handle threshold=0 case separately (unlikely).
    signal thresholdMinusOne <== threshold - 1;
    component ageCheck = LessThan(32);
    ageCheck.in[0] <== thresholdMinusOne;
    ageCheck.in[1] <== diff;
    // If ageCheck.out == 1, then threshold-1 < diff, so diff >= threshold.
    isAdult <== ageCheck.out;
}

component main {public [today, threshold]} = AgeVerification();

4.1 Explanation of Constraints

Birthdate validity: The validBirth signal is the sum of the LessThan output and the equality output. Since both are boolean (0 or 1), their sum is 1 only if exactly one condition holds (or both if birthDay < today and birthDay == today is impossible). Actually if birthDay < today, LessThan gives 1, IsEqual gives 0, sum=1. If birthDay == today, LessThan gives 0, IsEqual gives 1, sum=1. If birthDay > today (future), both give 0, sum=0, and the constraint validBirth === 1 fails. Perfect.

Difference: The subtraction today - birthDay is safe because we have already proven birthDay <= today, so no underflow.

Age threshold: To check diff >= threshold, we use the trick: threshold - 1 < diff. If threshold - 1 is strictly less than diff, then diff is at least threshold. The LessThan component outputs 1 if the first input is less than the second. So we set ageCheck.in[0] = thresholdMinusOne and ageCheck.in[1] = diff. If thresholdMinusOne < diff, out=1, and isAdult = 1. If diff == threshold - 1, then thresholdMinusOne is not less, out=0, isAdult=0 - exactly what we want (age is not yet 18 if exactly 17 years and 364 days).

4.3 Complete snarkjs Workflow - Step-by-Step Commands

Now that you understand the circuit and its constraints, let us walk through the entire workflow from source code to a verified proof. You will need Node.js (v16+), Circom (v2.0+), and snarkjs installed globally. Open a terminal in your project directory and follow each step in order.

Step 1
Compile the Circom circuit
The compiler produces two critical files: the R1CS (constraint system) and the WASM module (witness generator). The --sym flag generates a symbol file for debugging.
Terminal - Compile the circuit
# Create a project directory
mkdir age-verify && cd age-verify

# Save your circuit code as ageVerification.circom
# Then compile it:
circom ageVerification.circom --r1cs --wasm --sym -o build/

# Expected output:
# template instances: 3
# non-linear constraints: ~250
# linear constraints: 0
# public inputs: 2
# private inputs: 1
# public outputs: 1
# wires: ~280
# labels: ~300
Step 2
Powers of Tau ceremony (Phase 1 trusted setup)
Groth16 requires a trusted setup. Phase 1 is circuit-independent - it generates universal parameters. For production, you would use an existing community ceremony (e.g., Hermez's or Zcash's). For development, we generate our own. The parameter 12 means up to 2^12 = 4096 constraints.
Terminal - Powers of Tau (Phase 1)
# Start a new Powers of Tau ceremony (BN128 curve, 2^12 max constraints)
snarkjs powersoftau new bn128 12 pot12_0000.ptau -v

# Contribute randomness (in production, multiple parties contribute)
snarkjs powersoftau contribute pot12_0000.ptau pot12_0001.ptau \
  --name="First contribution" -v -e="some random text for entropy"

# Prepare Phase 2
snarkjs powersoftau prepare phase2 pot12_0001.ptau pot12_final.ptau -v
Step 3
Circuit-specific setup (Phase 2 trusted setup)
Phase 2 is specific to your circuit. It produces the proving key (zkey) and the verification key. In production, this phase also requires multiple contributors to ensure no single party can forge proofs.
Terminal - Phase 2 Setup
# Generate the initial zkey (circuit-specific)
snarkjs groth16 setup build/ageVerification.r1cs pot12_final.ptau \
  ageVerification_0000.zkey

# Contribute to Phase 2
snarkjs zkey contribute ageVerification_0000.zkey ageVerification_0001.zkey \
  --name="Phase2 contribution" -v -e="more random entropy"

# Export the final verification key (JSON)
snarkjs zkey export verificationkey ageVerification_0001.zkey \
  verification_key.json
Step 4
Create the input file (input.json)
The input file contains both private and public inputs. The prover knows all values; the verifier only sees the public ones after verification. Here, we compute the birthdate and today's date as days since Unix epoch.
input.json - Example for someone born 15-Aug-2000
{
  "birthDay": 11184,
  "today": 20454,
  "threshold": 6570
}

// How to compute these values:
// birthDay: Date(2000-08-15) => days since 1970-01-01 = 11184
// today:    Date(2026-01-15) => days since 1970-01-01 = 20454
// threshold: 18 years * 365 days = 6570 (simplified)
// diff = 20454 - 11184 = 9270 days (about 25.4 years) => isAdult = 1
🛠️ Computing Days Since Epoch in JavaScript
// Quick helper to convert a date to days-since-epoch
function dateToDays(dateStr) {
  const ms = new Date(dateStr).getTime();
  return Math.floor(ms / (1000 * 60 * 60 * 24));
}

console.log(dateToDays("2000-08-15")); // 11184
console.log(dateToDays("2026-01-15")); // 20454
Step 5
Generate the witness
The WASM module produced during compilation computes all intermediate signals from your inputs. The output is a binary .wtns file containing every signal value (the "witness") that satisfies the R1CS constraints.
Terminal - Witness Generation
# Generate the witness using the WASM module
node build/ageVerification_js/generate_witness.js \
  build/ageVerification_js/ageVerification.wasm \
  input.json \
  witness.wtns

# If successful, no error is printed and witness.wtns is created.
# If birthDay > today, the witness generation FAILS (constraint violation).
Step 6
Generate the Groth16 proof
The prover combines the witness and the proving key to produce the proof. The output is a proof.json file (the cryptographic proof) and a public.json file (the public inputs and outputs).
Terminal - Proof Generation
# Generate the proof
snarkjs groth16 prove ageVerification_0001.zkey witness.wtns \
  proof.json public.json

# Inspect the public outputs
cat public.json
# Expected output: ["1", "20454", "6570"]
# The "1" is isAdult (the circuit's output signal).
# "20454" is today, "6570" is threshold - both public inputs.
Step 7
Verify the proof (off-chain)
Before deploying on-chain, you can verify the proof locally using snarkjs. This confirms the proof is valid against the verification key and the public inputs.
Terminal - Verification
# Verify the proof off-chain
snarkjs groth16 verify verification_key.json public.json proof.json

# Expected output:
# [INFO]  snarkJS: OK!

# If the proof is invalid (e.g., tampered), you get:
# [ERROR] snarkJS: Invalid proof
💡 What Happens If We Change birthDay?

Try modifying input.json to set birthDay to a recent date, say 19700 (about 2023). Now diff = 20454 - 19700 = 754 days, which is far less than the threshold of 6570. The witness generation will succeed, but isAdult will be 0. The proof will still be valid - it honestly proves "this person is NOT an adult." The circuit does not reject the proof; it simply outputs 0. The verifier's application logic must check that public.json[0] == "1" before granting access. This is a common source of bugs: verifying the proof is valid but forgetting to check the actual output value.

05   Integration with Indian Identity Systems

From Aadhaar XML to ZK Proof

The circuit above assumes birthDay is already extracted from a trusted credential. In practice, the user's Aadhaar XML (or DigiLocker VC) contains the birthdate in YYYY-MM-DD format. The SARAL wallet can parse this, convert it to days-since-epoch, and feed it into the circuit. Importantly, the user must also prove that the birthDay used in the circuit corresponds to the birthdate in the credential - otherwise they could input any birthdate. This requires a linked proof: the circuit also takes as private input the Aadhaar VC and its signature, verifies the signature inside the circuit, and extracts the birthdate. That's more complex, but for simplicity, in this exercise we assume the wallet has already authenticated the credential, and we only prove age from the extracted birthdate. In production, you'd combine the VC verification and age proof into one circuit.

MST's SARAL protocol can store the birthdate VC in the wallet, and the wallet's WASMify module can handle the conversion and proof generation. The verifier contract on MST can be invoked by any dApp (bar, liquor shop, voting booth) with a simple call to verifyAgeProof(proof, pubInputs).

5B.2 Understanding the Auto-Generated Contract

The generated contract looks complex, but the interface is simple. Below is a simplified view of its structure with the key function that your dApp will call:

Simplified Solidity Verifier (AgeVerifier.sol)
// SPDX-License-Identifier: GPL-3.0
pragma solidity >=0.7.0 <0.9.0;

contract Groth16Verifier {
    // --- Verification key constants (auto-generated) ---
    uint256 constant alphax = 0x...; // Elliptic curve points
    uint256 constant alphay = 0x...; // from trusted setup
    // ... (dozens more constants for beta, gamma, delta, IC)

    // --- Pairing check function (internal) ---
    function _pairingProd4(...) internal view returns (bool) {
        // Uses EIP-197 precompiled contract for BN128 pairing
        // Gas cost: ~45,000 per pairing check
    }

    // --- THE FUNCTION YOUR DAPP CALLS ---
    function verifyProof(
        uint[2] calldata _pA,      // Proof point A
        uint[2][2] calldata _pB,   // Proof point B
        uint[2] calldata _pC,      // Proof point C
        uint[3] calldata _pubSignals // [isAdult, today, threshold]
    ) public view returns (bool) {
        // Checks that:
        // e(A, B) == e(alpha, beta) * e(IC, gamma) * e(C, delta)
        // where IC encodes the public inputs
        return _pairingProd4(...);
    }
}
🛠️ Calling the Verifier from Your dApp

In your dApp's front-end or back-end, you would call the verifier like this:

// JavaScript (ethers.js) - calling the verifier on MST
const verifier = new ethers.Contract(VERIFIER_ADDRESS, abi, provider);

// proof and publicSignals come from snarkjs output
const calldata = await snarkjs.groth16.exportSolidityCallData(
  proof, publicSignals
);

// Parse the calldata
const [pA, pB, pC, pubSignals] = JSON.parse("[" + calldata + "]");

// Call the on-chain verifier
const isValid = await verifier.verifyProof(pA, pB, pC, pubSignals);

if (isValid && pubSignals[0] === "1") {
  console.log("Age verified! Grant access.");
} else {
  console.log("Verification failed or user is underage.");
}

The contract is stateless (it only reads the verification key constants and computes a pairing check), so it requires no constructor arguments and minimal gas to deploy. Once deployed, any number of dApps can reference the same verifier contract address - you do not need a separate contract for each bar, liquor shop, or voting booth.

Deploy to MST testnet first
Use Hardhat's deploy script or Foundry's forge create to deploy the contract to MST's testnet. Record the contract address for your dApp configuration.

Verify and register

💡 Gas Cost Considerations

On-chain Groth16 verification on EVM-compatible chains like MST costs approximately 230,000-250,000 gas. This is because the verifier performs four elliptic curve pairing operations using the EIP-197 precompile. At typical MST gas prices, this translates to a fraction of a rupee per verification - far cheaper than the manual ID check process it replaces. For comparison, a simple ERC-20 token transfer costs about 65,000 gas. So a ZK proof verification is roughly 4x the cost of a token transfer, which is remarkably affordable for the privacy guarantees it provides.

05-C   Extending the Circuit

Beyond Age - Other Privacy-Preserving Proofs

The age verification circuit is a foundation that can be adapted for many other privacy-preserving eligibility checks. The core pattern - "prove a private value satisfies a public condition without revealing the value" - applies broadly across Indian governance, finance, and social welfare scenarios. Below are four concrete extensions, each building on the same Circom + snarkjs + MST stack.

💰
Income Bracket Proof
Prove your annual income falls within a range (e.g., below Rs. 8 lakh for EWS reservation) without revealing the exact figure. The circuit takes the income as a private input and checks income < upperBound and income >= lowerBound. The Verifiable Credential could be an ITR (Income Tax Return) document issued via DigiLocker. This eliminates the need for income certificate forgery checks that plague government subsidy programs - a problem that costs the Indian exchequer an estimated Rs. 1,500 crore annually in fraudulent claims.
🏠
Residency Proof
Prove you reside in a specific state or district without revealing your full address. The circuit takes the state code (or district code) as a private input and checks it against a set of valid codes. This is useful for state-specific subsidies (e.g., Karnataka's Gruha Lakshmi scheme), college admissions with domicile requirements, or even election eligibility. The proof reveals only "this person is a resident of Karnataka" without exposing their exact locality, PIN code, or house number - crucial for protecting against stalking and targeted crime.
📜
Credential Validity Proof
Prove that a professional credential (medical licence, driving licence, engineering degree) is currently valid without revealing the credential number or holder's identity. The circuit checks that the credential's expiry date is in the future and that the issuer's digital signature is valid. This is especially relevant for India's healthcare sector, where the National Medical Commission could issue ZK-verifiable credentials to doctors, allowing hospitals to confirm a doctor's registration status without accessing the full NMC database.
🗳️
Voting Eligibility Proof
Prove you are a registered voter in a constituency without revealing your voter ID or EPIC number. The circuit takes the voter roll Merkle proof as a private input and verifies membership in the constituency's voter set. Combined with a nullifier (a unique hash derived from your identity), the system can ensure one-person-one-vote while maintaining ballot secrecy. This is the foundation of ZK voting systems like MACI (Minimum Anti-Collusion Infrastructure), adapted for Indian electoral requirements.

💡 The Groth16 Sweet Spot

Notice the remarkable property of Groth16: proof size and verification cost are constant, no matter how complex the circuit. Whether you are proving age (250 constraints) or verifying a full credential signature (55,000 constraints), the proof is always 192 bytes and verification costs roughly 230,000-250,000 gas. The only thing that changes is the prover's computation time. This makes Groth16 ideal for on-chain verification where gas costs are the bottleneck. The trade-off is the trusted setup requirement - each circuit needs its own ceremony. Newer systems like PLONK offer a universal setup (one ceremony for all circuits) but produce slightly larger proofs (~400-600 bytes) and cost more gas to verify (~300,000-350,000 gas).

5D.1 Mobile Performance Considerations

In the Indian context, many users will generate proofs on mobile devices (Android smartphones in the Rs. 8,000-15,000 range). Here, proof generation for the basic age circuit takes approximately 2-4 seconds using the WASM prover in a WebView or React Native app. For the full VC-bound proof with signature verification, mobile proof generation can take 30-60 seconds on mid-range devices. To optimise for mobile:

1
Use the native C++ prover via rapidsnark
The rapidsnark library is a C++ implementation of the Groth16 prover that is 5-10x faster than the JavaScript/WASM prover. It can be compiled for Android (ARM64) and integrated into the SARAL wallet as a native module. With rapidsnark, even the full VC-bound proof completes in under 5 seconds on a mid-range phone.
2
Pre-compute the witness off the critical path
The SARAL wallet can compute the witness (all intermediate signal values) in the background as soon as the user opens the app, before they even scan the bar's QR code. When the actual verification request arrives, only the proof generation step remains, cutting perceived latency in half.
3
Cache the proving key in local storage
The proving key (.zkey file) for the age verification circuit is approximately 15-20 MB. Downloading it every time would be impractical on Indian mobile networks. The SARAL wallet should download the key once during setup and cache it locally. Updates to the key (if the circuit changes) can be handled via a versioned CDN.

5D.2 Proof Aggregation for High-Throughput Scenarios

Consider a large music festival in Goa with 10,000 attendees, each needing age verification at the gate. Verifying 10,000 individual proofs on-chain would cost roughly 2.3 billion gas - impractical even on MST. Proof aggregation solves this by combining multiple proofs into a single proof that can be verified in one on-chain transaction. The aggregated proof says "all 10,000 attendees are adults" and costs the same ~250,000 gas as a single verification.

Aggregation works by creating a "recursive" circuit: a circuit that verifies other proofs. The aggregator collects individual proofs off-chain, feeds them into the recursive circuit, and produces one combined proof. Tools like SnarkPack and Halo2 support this pattern. While more complex to implement, proof aggregation is essential for any deployment that must handle hundreds or thousands of verifications per hour.

06   Common Pitfalls & Best Practices

What Can Go Wrong - and How to Avoid It

⚠️
Missing equality check
If you only use LessThan to check birthDay < today, a user born exactly at midnight today (age 0) could pass as adult. Always include an equality check or use a combination to allow birthDate <= today.
⚠️
Using today provided by prover
If the verifier doesn't check that today matches the actual current date, a prover can supply a future date and appear older. Always make today a public input that the verifier provides or checks against a trusted source.
⚠️
Integer overflow
Days-since-epoch for a birthdate in 1970 is small, but for safety use 64-bit numbers (the LessThan template parameter). Circom signals are field elements (prime p ~ 2^254), so overflow is not a risk, but bit-width affects constraints.
⚠️
Forgetting to check the output signal
A valid proof does not mean the user is an adult. The proof might honestly say isAdult = 0. Always check that publicSignals[0] == 1 in your application logic after verifying the proof. Many developers verify the proof but forget to inspect the output - a critical security bug.
⚠️
Reusing a ceremony across circuits
Phase 1 (Powers of Tau) is universal, but Phase 2 is circuit-specific. If you modify even one constraint in your Circom code, you must re-run Phase 2 to generate a new proving key. Reusing an old zkey with a modified circuit will produce invalid proofs or, worse, proofs that verify but are unsound.
⚠️
Leaking information through timing
If proof generation takes noticeably different amounts of time for different birthdates (e.g., faster for older people), an observer could estimate the user's age from the generation time. In practice, Groth16 proof generation time depends on the number of constraints (which is fixed for a given circuit), not on the input values. However, witness generation in JavaScript might have timing side-channels. Use constant-time libraries when possible.
07   Assessment Preparation

Top Facts - Know These Cold

01
Range proof:
Proving that a hidden number lies within a range without revealing it. Age verification uses a range proof: age ∈ [18, ∞).
02
LessThan template:
From circomlib, checks a < b for n-bit numbers. Used to build comparisons. For >=, use (threshold-1) < x.
03
Public vs Private inputs:
Birthdate is private; today and threshold are public. The verifier must enforce today's value. Public inputs are visible to the verifier but not necessarily to the blockchain (unless output).
04
Groth16 proof:
~200 bytes, verification in ~1ms. Requires a trusted setup ceremony. MST's verifier contract uses pre-compiled pairing checks.
05
Binding to VC:
In production, the circuit also verifies the VC signature to ensure the birthdate is from an authorised issuer (UIDAI). This can be done by hashing the VC and checking against the issuer's public key.
06
R1CS constraint count:
The age verification circuit produces approximately 250 R1CS constraints. Each LessThan(32) template generates roughly 100 constraints (32 bit-decomposition checks plus auxiliary constraints). More constraints mean a larger proving key and slower proof generation, but verification cost remains constant.
07
Trusted setup phases:
Phase 1 (Powers of Tau) is circuit-independent and can be reused. Phase 2 is circuit-specific and must be re-run if the circuit changes. Both phases require multiple honest participants to ensure soundness. The "1-of-N" security assumption means only one participant needs to be honest.
08
Proof size is constant:
Groth16 proofs are always 192 bytes (three BN128 curve points: A, B, C). This is true regardless of circuit complexity - a 250-constraint circuit and a 1-million-constraint circuit produce identical proof sizes. This constant size is Groth16's defining advantage over other proof systems.
08   Glossary

Key Terms - Sub-Module 16.5

Range Proof
A cryptographic proof that a secret number lies within a specific range without revealing the number itself. Age verification is a one-sided range proof (≥ threshold).
LessThan Template
A Circom template that proves a < b for two n-bit numbers. Foundation of inequality checks in ZK circuits.
snarkjs
JavaScript/WASM library for trusted setup, proof generation, and verification of Groth16 zk-SNARKs, compatible with Circom outputs.
Public Input
A value that is known to both prover and verifier. In age verification, today and threshold are public; birthdate is private.
Groth16
A widely used zk-SNARK proving system with very small proofs (~200 bytes) and millisecond verification, requiring a circuit-specific trusted setup.
VC-Binding
The technique of proving that a ZK circuit's private inputs are derived from a valid Verifiable Credential, preventing input forgery.
R1CS (Rank-1 Constraint System)
The low-level mathematical representation of a ZK circuit. Every computation is flattened into constraints of the form A * B = C, where A, B, and C are linear combinations of circuit variables. The Circom compiler produces R1CS automatically from your high-level code.
Witness Generation
The process of computing all intermediate signal values in a circuit from the provided inputs. The WASM module produced by the Circom compiler performs this computation. The resulting witness file (.wtns) contains every value needed to satisfy the R1CS constraints.
Powers of Tau
Phase 1 of the Groth16 trusted setup ceremony. It generates universal parameters (toxic waste) that are circuit-independent. Multiple participants contribute randomness; as long as at least one participant is honest and destroys their secret, the ceremony is secure. Community ceremonies (e.g., Hermez, Zcash) have had thousands of contributors.
Phase 2 Ceremony
The circuit-specific portion of the Groth16 trusted setup. It takes the Phase 1 output and the circuit's R1CS to produce the proving key (zkey) and verification key. Like Phase 1, multiple contributors enhance security. Each new circuit requires its own Phase 2 ceremony.
Verifier Contract
A Solidity smart contract auto-generated by snarkjs that can verify Groth16 proofs on-chain. It embeds the verification key as constants and uses EIP-197 precompiled contracts for BN128 elliptic curve pairing checks. Verification costs approximately 230,000-250,000 gas on EVM chains.
Proof Aggregation
The technique of combining multiple individual ZK proofs into a single proof that can be verified in one operation. Uses recursive circuits (circuits that verify other proofs) to achieve constant verification cost regardless of how many original proofs are aggregated. Essential for high-throughput deployments.
