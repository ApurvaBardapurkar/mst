04   Step 1 - Generate the Smart Contract with OpenZeppelin Wizard

No Need to Write ERC-721 from Scratch

The OpenZeppelin Contract Wizard is a web-based tool that generates secure, audited smart contract code based on the options you select. It's the safest and fastest way to create an NFT contract for any EVM chain, including MST.

Open https://wizard.openzeppelin.com in your browser. Select the ERC-721 tab. You'll see a visual interface where you can toggle features on and off. Here is the configuration we'll use for our collection:

Name & Symbol: Set the name to "MST Collection" and the symbol to "MSTC".

Base URI: Leave this empty for now - we'll set it after uploading to IPFS.

Mintable: Enable this. It adds a safeMint function that only the contract owner can call.

Auto-increment IDs: Enable this. Each NFT will get a sequential token ID starting from 0.

Enumerable: Enable this. It adds functions to list all token IDs and query total supply, which marketplaces need.

URI Storage: Enable this. It adds a function to set the base URI for all token metadata.

Royalties (ERC-2981): Enable this. Set a royalty percentage (e.g., 5%) and a royalty receiver address (your wallet). This means you'll earn 5% on every secondary sale forever.

Ownable: Enable this. It restricts minting and administrative functions to the contract owner.

Pausable: Enable this (optional). It lets you pause minting in an emergency.

Click "Copy to Clipboard" and save the generated code as contracts/MyCollection.sol in your Hardhat project. The wizard produces a complete, production-ready contract with all the features you selected.

// contracts/MyCollection.sol (generated by OpenZeppelin Wizard)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/token/ERC721/extensions/ERC721Enumerable.sol";
import "@openzeppelin/contracts/token/ERC721/extensions/ERC721URIStorage.sol";
import "@openzeppelin/contracts/token/ERC721/extensions/ERC721Pausable.sol";
import "@openzeppelin/contracts/access/Ownable.sol";
import "@openzeppelin/contracts/token/common/ERC2981.sol";

contract MyCollection is ERC721, ERC721Enumerable, ERC721URIStorage, ERC721Pausable, Ownable, ERC2981 {
    uint256 private _nextTokenId;

    constructor(address initialOwner, address royaltyReceiver)
        ERC721("MST Collection", "MSTC")
        Ownable(initialOwner)
    {
        _setDefaultRoyalty(royaltyReceiver, 500); // 5% royalty
    }

    function safeMint(address to, string memory uri) public onlyOwner {
        uint256 tokenId = _nextTokenId++;
        _safeMint(to, tokenId);
        _setTokenURI(tokenId, uri);
    }

    function pause() public onlyOwner { _pause(); }
    function unpause() public onlyOwner { _unpause(); }

    // Overrides required by Solidity
    function _update(address to, uint256 tokenId, address auth)
        internal override(ERC721, ERC721Enumerable, ERC721Pausable)
        returns (address)
    { return super._update(to, tokenId, auth); }

    function _increaseBalance(address account, uint128 value)
        internal override(ERC721, ERC721Enumerable)
    { super._increaseBalance(account, value); }

    function tokenURI(uint256 tokenId)
        public view override(ERC721, ERC721URIStorage)
        returns (string memory)
    { return super.tokenURI(tokenId); }

    function supportsInterface(bytes4 interfaceId)
        public view override(ERC721, ERC721Enumerable, ERC721URIStorage, ERC2981)
        returns (bool)
    { return super.supportsInterface(interfaceId); }
}
04A   Deep Dive - Understanding Each OpenZeppelin Import

What Every Import Does and Why It Matters

The contract generated by the OpenZeppelin Wizard imports several modules, and each one adds a specific capability to your NFT collection. Understanding these imports is critical for customising your contract later and for debugging issues during development. Think of each import as a specialised department in a company - each handles a specific responsibility so the main contract stays clean and focused.

💡 Analogy

Imagine you are building a cricket team. The base ERC-721 contract is like having a captain who can manage basic play. Each extension is like adding a specialist - a wicket-keeper (Enumerable for tracking), a batting coach (URIStorage for metadata management), a financial advisor (ERC2981 for royalties), and a team manager (Ownable for access control). You could play without them, but the team would be incomplete.

1️⃣
ERC721 (Base Contract)
This is the core NFT standard. It provides the fundamental functions: ownerOf() to check who owns a token, transferFrom() to send a token to another address, approve() to let someone else transfer your token, and balanceOf() to check how many tokens an address holds. Every ERC-721 contract must implement this base. Without it, your contract is not recognised as an NFT by wallets, marketplaces, or block explorers.
2️⃣
ERC721Enumerable
This extension adds the ability to list all tokens in the collection and all tokens owned by a specific address. It provides three key functions: totalSupply() returns the total number of minted NFTs, tokenByIndex(index) returns the token ID at a given position in the global list, and tokenOfOwnerByIndex(owner, index) returns a specific token owned by an address. Marketplaces like OpenSea rely on these functions to display your full collection. Without Enumerable, a marketplace would have no efficient way to discover which tokens exist or who owns them. The trade-off is higher gas costs during minting (roughly 20-30% more), because the contract must maintain internal arrays for tracking.
3️⃣
ERC721URIStorage
This extension allows each token to have its own unique metadata URI stored on-chain. The standard ERC-721 computes token URIs by appending the token ID to a base URI (e.g., ipfs://Qm.../0). With URIStorage, you can set a completely custom URI for each token using _setTokenURI(tokenId, uri). This is useful when your metadata files don't follow a simple numeric naming pattern, or when different tokens point to different IPFS folders. For a standard collection where all metadata is in one folder with files named 0.json, 1.json, etc., you could use a base URI instead - but URIStorage gives you maximum flexibility.
4️⃣
ERC721Pausable
This extension lets the contract owner pause and unpause all token transfers (including minting). When paused, no tokens can be minted, transferred, or burned. This is an emergency mechanism - for example, if you discover a bug in your metadata, you can pause minting while you fix it. In the Indian startup ecosystem, this is similar to how the RBI can temporarily halt transactions on a payment platform if a security issue is detected. Use it sparingly; a permanently paused contract will frustrate your community.
5️⃣
ERC2981 (Royalty Standard)
This standard defines a universal way for NFT contracts to signal royalty information. When a marketplace queries your contract with royaltyInfo(tokenId, salePrice), your contract returns two values: the royalty receiver's address and the royalty amount (calculated as a percentage of the sale price). The _setDefaultRoyalty(receiver, feeNumerator) function sets a global royalty for all tokens. The fee numerator uses a denominator of 10,000 - so 500 means 500/10,000 = 5%. You can also set per-token royalties using _setTokenRoyalty(). Note that royalty enforcement depends on the marketplace honouring ERC-2981 - most major marketplaces do, but it is not enforced at the protocol level.
6️⃣
Ownable
This module restricts certain functions to a single "owner" address. The onlyOwner modifier ensures that only the owner can call functions like safeMint, pause, and unpause. Ownership can be transferred using transferOwnership(newOwner) and renounced entirely using renounceOwnership(). For a public mint where anyone should be able to mint, you would remove the onlyOwner modifier from the mint function and add a payable requirement instead.
⚠️ Override Conflicts

When you inherit from multiple OpenZeppelin contracts, some of them override the same internal functions. For example, both ERC721Enumerable and ERC721Pausable override the _update function. Solidity requires you to explicitly resolve these conflicts by creating your own override that calls super._update(). The OpenZeppelin Wizard handles this automatically, but if you add extensions manually later, you must add the corresponding overrides yourself or the compiler will throw an error.

06   Step 3 - Prepare Metadata and Artwork

Every NFT Needs a JSON File and an Image

For a collection of 10 NFTs, create a folder metadata/ and a folder images/. For each token ID (0 through 9), you need:

1. An image file (e.g., 0.png, 1.png, etc.) in the images/ folder.

2. A JSON metadata file (e.g., 0.json, 1.json) that references the image and describes the NFT.

{
    "name": "MST Collection #0",
    "description": "The very first NFT in the MST Collection – a symbol of your journey into Web3.",
    "image": "ipfs://QmYourImagesCID/0.png",
    "attributes": [
        { "trait_type": "Background", "value": "Midnight Blue" },
        { "trait_type": "Rarity", "value": "Common" }
    ]
}

After uploading all images to IPFS, you'll get a single CID for the folder (e.g., QmYourImagesCID). Replace the image URL in every JSON file with ipfs://QmYourImagesCID/0.png (changing the filename for each). Then upload the entire metadata/ folder to IPFS, which gives you a second CID for the metadata folder.

Your base URI will be ipfs://QmYourMetadataCID/. The contract will construct the full token URI as ipfs://QmYourMetadataCID/0.json, 1.json, etc.

06A   Deep Dive - Metadata Best Practices

Designing Metadata That Marketplaces Love

Metadata is the soul of your NFT. The smart contract only stores a pointer (the token URI) - the actual name, description, image, and attributes live in the JSON metadata file. Poorly structured metadata leads to broken images on marketplaces, missing attributes in rarity tools, and a bad collector experience. Here are the best practices every NFT creator should follow.

Image Format and Size Guidelines

Your NFT images need to balance quality with loading speed. Marketplaces display images at various sizes - from 300x300 thumbnails in grid views to full-resolution on the detail page. Follow these recommendations:

Format Best For Max Size Notes
PNG Pixel art, illustrations, graphics with transparency 5 MB Lossless compression; preferred for most NFT art
JPEG Photographs, painterly art 5 MB Lossy but smaller files; no transparency
SVG Fully on-chain generative art 1 MB Vector; scales infinitely; can be stored on-chain
GIF / MP4 Animated art, video NFTs 50 MB Use animation_url field in metadata for video

For a standard collection, use PNG at 2000x2000 pixels. This gives crisp rendering at any display size. Keep individual files under 5 MB to ensure fast IPFS retrieval. If your collection is generative (like Bored Apes or CryptoPunks), generate all images at the same resolution for consistency.

Attribute Design for Rarity

Attributes (also called "traits" or "properties") are the key-value pairs in your metadata that describe an NFT's visual or conceptual properties. Rarity tools like Rarity Sniper and HowRare.is use these attributes to calculate rarity scores. Collectors pay premiums for rare trait combinations, so thoughtful attribute design directly impacts your collection's value.

💡 Indian Context Example

Imagine you are creating a collection called "Chai Wallahs of India" - 1,000 unique NFTs of illustrated chai sellers from different Indian cities. Your trait categories might include: City (Mumbai, Delhi, Kolkata, Chennai, Jaipur - with Mumbai being most common and Jaipur being rarest), Chai Type (Masala, Cutting, Irani, Kulhad, Butter - with Butter being legendary rare), Background (Street, Station, Office, Rooftop, Heritage Monument), and Accessories (Newspaper, Radio, Cricket Bat, Smartphone, None). Each trait has a distribution: common traits appear in 30-40% of NFTs, uncommon in 15-25%, rare in 5-10%, and legendary in 1-3%.

Trait Distribution Planning

Before generating your collection, plan the distribution of each trait value across your total supply. A well-planned distribution creates a natural rarity hierarchy that collectors find exciting. Here is a framework you can follow:

📊
Common Traits (30-40% occurrence)
These are the baseline traits that most NFTs share. In a 1,000-item collection, a common background might appear on 350 NFTs. They anchor the collection's visual identity and make rare traits stand out by contrast.
📊
Uncommon Traits (15-25% occurrence)
These traits are noticeably less frequent. They give collectors a sense of having something slightly special without being extremely rare. Example: in a collection of 500, an uncommon eye colour might appear on 100 NFTs.
📊
Rare Traits (5-10% occurrence)
Rare traits are the ones collectors actively hunt for. They significantly boost an NFT's rarity score. In a 1,000-item collection, a rare accessory might appear on only 50-80 NFTs. These often become the "floor price drivers" on secondary markets.
📊
Legendary Traits (1-3% occurrence)
The crown jewels of your collection. Legendary traits appear on fewer than 30 NFTs in a 1,000-item set. Some creators make certain traits appear on only 1 NFT (1-of-1 traits). These command the highest secondary market prices and become status symbols in your community.
⚠️ Common Metadata Mistakes

Inconsistent trait names: Using "Background Color" in some files and "Background" in others will cause rarity tools to treat them as different categories. Always use identical capitalisation and spelling across all metadata files.

Missing image field: If the image field is missing or points to a broken IPFS link, marketplaces will show a blank placeholder. Always verify that every CID resolves before deploying.

Numeric traits without display_type: If you have a trait like "Power Level: 85", add "display_type": "number" so marketplaces render it as a progress bar rather than plain text.

08A   Deep Dive - Contract Verification on MSTScan

Proving Your Code Is What It Claims to Be

Contract verification is the process of uploading your Solidity source code to a block explorer (MSTScan) so that anyone can read, audit, and interact with your contract directly from the explorer. Without verification, your contract appears as raw bytecode on MSTScan - a wall of hexadecimal characters that no one can understand. A verified contract shows the full Solidity source, making it transparent and trustworthy.

💡 Why Verification Matters

In the Indian context, think of contract verification like a company's MCA (Ministry of Corporate Affairs) registration. Anyone can look up a registered company's details - directors, address, articles of association. Similarly, a verified smart contract lets anyone inspect exactly what the code does. Collectors are far more likely to mint from a verified contract because they can confirm there are no hidden fees, no backdoor admin functions, and that royalties are set correctly. Unverified contracts are red flags in the NFT space.

Verification Using Hardhat

The Hardhat verification plugin submits your source code and compiler settings to MSTScan's API. If the compiled bytecode matches what is deployed on-chain, the contract is marked as verified. Run the following command:

npx hardhat verify --network mst_testnet YOUR_CONTRACT_ADDRESS "CONSTRUCTOR_ARG_1" "CONSTRUCTOR_ARG_2"

For our MyCollection contract, the constructor takes two arguments: initialOwner and royaltyReceiver (both are your deployer address). So the command would be:

npx hardhat verify --network mst_testnet 0xYourContractAddress "0xYourWalletAddress" "0xYourWalletAddress"

Common Verification Errors and Fixes

Error Message Cause Fix
Constructor arguments do not match The arguments you passed to the verify command differ from what was used during deployment Check your deploy script and pass the exact same arguments in the exact same order
Bytecode does not match You modified the contract source after deployment, or the compiler version differs Ensure hardhat.config.js specifies the exact Solidity version used during deployment (e.g., 0.8.20). Do not change the source code after deploying
Contract already verified You already verified this contract address No action needed - your contract is already verified. Check it on MSTScan
API key missing or invalid MSTScan requires an API key for verification Register on MSTScan, generate an API key, and add it to your hardhat.config.js under etherscan.apiKey

After successful verification, visit your contract on MSTScan. You will see a green checkmark next to the contract name, and tabs for "Read Contract" and "Write Contract" will appear. These tabs let anyone interact with your contract directly from the browser - minting NFTs, checking total supply, or reading royalty information - without needing a frontend.

09A   Deep Dive - Making the Minting Experience User-Friendly

From Functional to Delightful - UX Best Practices for Minting Pages

A minting page that works is not the same as a minting page that people enjoy using. The difference between a sold-out collection and an abandoned one often comes down to user experience. Collectors - especially first-time crypto users - need clear feedback at every step. A confusing interface causes people to abandon the mint, share negative reviews, and lose trust in your project. Here are the critical UX patterns every minting frontend should implement.

Loading States and Transaction Feedback

Blockchain transactions are not instant. After a user clicks "Mint", the transaction must be signed in the wallet, submitted to the network, included in a block, and confirmed. This process can take anywhere from 5 seconds to 2 minutes depending on network congestion. During this time, your UI must communicate exactly what is happening.

⏳
Stage 1: Waiting for Wallet Approval
Show "Waiting for wallet confirmation..." with a subtle spinner. The user's MST BridgeKey popup is open - they need to click "Confirm". If they take too long (say, 60 seconds), show a gentle reminder: "Please check your wallet extension to approve the transaction."
📡
Stage 2: Transaction Submitted
Show "Transaction submitted! Waiting for confirmation..." and display the transaction hash as a clickable link to MSTScan. This lets the user verify the transaction is real and track its progress independently.
✅
Stage 3: Confirmed
Show a success animation with confetti or a checkmark. Display the minted NFT's image (fetch it from IPFS using the token URI) and a link to view it on MSTScan. Update the "Total Minted" counter in real time. This celebratory moment builds excitement and encourages sharing on social media.

Error Handling Patterns

Things will go wrong. The user might reject the transaction, run out of gas, try to mint when the supply is exhausted, or lose their internet connection mid-transaction. Good error handling turns frustrating failures into recoverable moments.

⚠️ Common Errors and User-Friendly Messages

User rejected transaction: Instead of showing the raw error "user rejected transaction", display "You cancelled the transaction. Click Mint again when you're ready - no gas was spent."

Insufficient funds: Display "Your wallet doesn't have enough MST to cover the mint price plus gas fees. You need approximately X MST." and link to the MST testnet faucet.

Supply exhausted: Replace the Mint button with "Sold Out!" and link to the secondary marketplace where collectors can buy from other holders.

Network error: Show "Network connection lost. Please check your internet and try again." with a retry button.

Wallet Switching and Network Detection

Users frequently have multiple wallet accounts or are connected to the wrong network. Your frontend should handle both scenarios gracefully. Use the window.ethereum events to detect account and network changes in real time:

// Detect network changes
window.ethereum.on('chainChanged', (chainId) => {
    if (parseInt(chainId, 16) !== 4545) {
        alert('Please switch to MST Testnet (Chain ID: 4545)');
    }
    window.location.reload();
});

// Prompt user to add MST Testnet if not configured
async function addMSTNetwork() {
    await window.ethereum.request({
        method: 'wallet_addEthereumChain',
        params: [{
            chainId: '0x11C1',
            chainName: 'MST Testnet',
            rpcUrls: ['https://testnetrpc.mstblockchain.com'],
            nativeCurrency: { name: 'MST', symbol: 'MST', decimals: 18 }
        }]
    });
}

Mobile Responsiveness

A significant portion of NFT collectors browse and mint from mobile devices. In India, mobile internet usage exceeds 75% of total web traffic, making mobile optimisation essential. Your minting page should use responsive CSS (flexbox or grid with media queries), have touch-friendly buttons (minimum 48x48 pixel tap targets), and detect mobile wallet apps like MST BridgeKey Mobile. On mobile browsers without a wallet extension, display a deep link to open the MST BridgeKey app directly, or show a QR code that the user can scan with their wallet app.

10A   Gas Optimisation Tips for NFT Contracts

Reduce Costs, Improve Speed, Delight Your Minters

Gas fees are one of the biggest pain points for NFT collectors. On high-traffic EVM chains, minting a single NFT can cost ₹500-₹2,000 in gas during peak times. While MST Blockchain has lower gas costs than Ethereum mainnet, optimising your contract still matters - it makes minting cheaper, faster, and more accessible. Here are the most impactful optimisation patterns used by professional NFT developers.

Batch Minting

Standard ERC-721 contracts mint one token per transaction. If a collector wants to mint 5 NFTs, they must submit 5 separate transactions, paying gas for each. Batch minting allows a user to mint multiple tokens in a single transaction, sharing the fixed gas overhead (21,000 gas for the transaction itself) across all tokens. Here is a simple batch minting function you can add to your contract:

function batchMint(address to, string[] memory uris) public onlyOwner {
    for (uint256 i = 0; i < uris.length; i++) {
        uint256 tokenId = _nextTokenId++;
        _safeMint(to, tokenId);
        _setTokenURI(tokenId, uris[i]);
    }
}

With batch minting, minting 5 NFTs costs roughly 60-70% of what 5 individual mints would cost, because the base transaction fee and contract loading costs are paid only once.

The ERC721A Pattern

Developed by the Azuki team, ERC721A is an optimised implementation of ERC-721 that dramatically reduces gas costs for batch minting. The standard ERC-721 writes ownership data for every single token during minting. ERC721A uses a clever trick: it only writes ownership data for the first token in a batch, and uses lazy evaluation to infer ownership of subsequent tokens. This reduces the gas cost of minting 5 NFTs from approximately 5x the single-mint cost to just 1.1-1.3x.

💡 Real Numbers Comparison

On Ethereum mainnet (for reference), minting 5 NFTs with standard ERC-721 costs approximately 750,000 gas. With ERC721A, the same 5 mints cost approximately 180,000 gas - a 76% reduction. On MST Blockchain, the absolute costs are lower, but the percentage savings are similar. For a collection expecting high-volume mints (like a 10,000-item PFP collection), ERC721A can save your community lakhs of rupees in collective gas fees.

Lazy Minting

Lazy minting (also called "gasless minting" or "mint on demand") is a pattern where the NFT is not actually minted on-chain until someone buys it. Instead of the creator paying gas to mint all 10,000 tokens upfront, the creator signs a voucher (a message containing the token metadata and price) off-chain. When a buyer decides to purchase, the buyer's transaction mints the token and pays the gas. This approach has three major advantages:

💰
Zero Upfront Cost for Creators
The creator never pays gas to mint. This is transformative for independent Indian artists who may not have the capital to mint a large collection upfront. An artist in Varanasi can list 1,000 NFTs without spending a single rupee on gas - the buyers pay the minting cost when they purchase.
🗑️
No Wasted Tokens
With eager minting, if you mint 1,000 tokens but only 200 sell, you've paid gas for 800 unsold tokens. With lazy minting, only tokens that are actually purchased get minted on-chain. No waste, no unnecessary blockchain bloat.
📋
Flexible Pricing
Since each voucher is signed independently, the creator can set different prices for different tokens, offer discounts to specific wallets (allowlists), or change prices over time without modifying the smart contract.

Other Quick Gas Savings

Beyond the major patterns above, several smaller optimisations can reduce gas costs by 10-30%:

Use uint96 instead of uint256 for token IDs when your collection will never exceed 79 billion tokens (which is every collection ever). Smaller integers can be packed into the same storage slot as other variables, saving gas.

Remove Enumerable if you don't need on-chain enumeration. The ERC721Enumerable extension adds significant storage overhead. If your frontend fetches token data from an indexer or subgraph rather than from the contract directly, you can skip this extension and save 20-30% on minting gas.

Use a merkle tree for allowlists instead of storing a mapping of allowed addresses on-chain. A merkle root is a single bytes32 value stored on-chain, while a mapping of 1,000 addresses would require 1,000 storage writes. With a merkle proof, users prove they are on the list by submitting a proof alongside their mint transaction.

12   Glossary

Key Terms - Sub-Module 12.5

OpenZeppelin Wizard
A web-based tool that generates secure, audited smart contract code based on selected features. Eliminates the need to write ERC-721 from scratch.
IPFS
InterPlanetary File System - a decentralised, peer-to-peer file storage network. NFT metadata and images should be stored on IPFS for true decentralisation.
Pinata
A free IPFS pinning service that ensures your files remain available on the IPFS network. Used to upload NFT images and metadata.
ERC-2981 (Royalties)
A standard that allows NFT creators to receive a percentage of every secondary sale. Marketplaces like OpenSea automatically enforce it.
Base URI
The common prefix for all token metadata URIs. If your base URI is ipfs://Qm.../, token 0's metadata is at ipfs://Qm.../0.json.
MST Testnet
A testing network for MST Blockchain where you can deploy contracts and mint NFTs without using real value. Chain ID: 4545.
MST BridgeKey
MST's non-custodial wallet (browser extension). Used to connect to dApps, sign transactions, and manage NFTs on MST.
Marketplace Listing
The act of offering an NFT for sale on a platform like OpenSea. Requires approving the marketplace to transfer the NFT and setting a price.
ERC721A
An optimised implementation of ERC-721 developed by the Azuki team that dramatically reduces gas costs for batch minting by using lazy ownership evaluation instead of writing data for every token individually.
Lazy Minting
A minting pattern where tokens are not created on-chain until a buyer purchases them. The creator signs off-chain vouchers, and the buyer's transaction handles the actual minting, eliminating upfront gas costs for the creator.
Metadata Standards
The JSON schema that NFT marketplaces expect for displaying NFTs correctly. Includes required fields like name, description, image, and optional fields like attributes, animation_url, and external_url.
Token Gate
A mechanism that restricts access to content, communities, events, or features based on NFT ownership. For example, only holders of a specific NFT collection can access a Discord channel, attend a virtual meetup, or unlock premium content on a website.
Reveal Mechanism
A technique where all NFTs in a collection initially display a placeholder image, and the actual artwork is revealed at a later date by updating the base URI. This builds anticipation and prevents snipers from cherry-picking rare tokens during mint.
Contract Verification
The process of submitting your Solidity source code to a block explorer (like MSTScan) to prove that the deployed bytecode matches the published source. Verified contracts show a green checkmark and allow direct interaction through the explorer interface.
