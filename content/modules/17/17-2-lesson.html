03   Background & Design Rationale

Why "Just Add a New Signature" Doesn't Work - and How MST Solved It

Most blockchain quantum-resistance proposals fall into one of two traps: either they require a disruptive hard fork that splits the community, or they ask every user to manually migrate funds to a new address type - a UX catastrophe that would lock millions of non-technical users out of their assets. MST's engineering team studied these failures and designed a backward-compatible, soft-fork-based migration that works with the existing consensus, wallet, and smart-contract infrastructure.

1
The Dual-Stack Transaction Format
MST's transaction structure was extended with an optional pq_signature field. Before the transition height, this field is ignored. After the transition height, validators require it and verify it against a registered post-quantum public key. This means old wallets continue to work until the transition, and new wallets can add the extra signature immediately.
2
No New Address Format - Just New Keys
Instead of creating a separate "pq1…" address type, MST binds a user's post-quantum public key to their existing ECDSA address through an on-chain key registry contract. When you deploy a SARAL wallet, a small identity contract is created that maps your address to both keys. Validators look up the pq key during signature verification - no address change, no UX disruption.
3
Governance-Scheduled Transition
The transition block height is set by MST's on-chain governance. This allows the community to react to real-world quantum developments - if a breakthrough is announced, the height can be accelerated. The default schedule assumes a conservative 18-month lead time from governance proposal to activation, with a 6-month hybrid window after activation.
🗝️ The Soft-Fork Principle

A soft fork means that the new rules are a subset of the old rules. Before the transition, hybrid transactions are valid under both old and new rules. After the transition, only hybrid (or fully pq) transactions are accepted by upgraded nodes. Non-upgraded nodes see blocks as valid because they still observe the old consensus rules, but they can't produce blocks without the pq signature. This prevents a chain split - a crucial requirement for financial networks.

💻 Key Registry Contract Interface (Solidity)
// Minimal identity contract that binds a user's post-quantum key
interface IPQKeyRegistry {
    function register(bytes calldata pqPublicKey) external;
    function getPQKey(address user) external view returns (bytes memory);
    event PQKeyRegistered(address indexed user, bytes pqPublicKey);
}

// Validator logic (simplified):
function verifyHybridSignature(address user, bytes32 txHash, bytes memory ecdsaSig, bytes memory pqSig) internal view {
    require(verifyECDSA(user, txHash, ecdsaSig), "ECDSA invalid");
    bytes memory pqKey = pqRegistry.getPQKey(user);
    require(pqKey.length > 0, "No PQ key registered");
    require(verifyDilithium(pqKey, txHash, pqSig), "Dilithium invalid");
}

What if a user hasn't registered a PQ key? After the transition height, unregistered addresses cannot initiate transactions. However, a "force-migrate" relay contract allows anyone to call a function that uses the user's pre-authorised migration allowance (set during SARAL wallet creation) to move assets to a fresh post-quantum address. This ensures no funds are permanently locked, even if the original owner lost access.

05   Gas Cost Analysis

Hybrid Transactions vs Classical: Understanding the Cost Overhead

One of the first questions any developer asks when confronted with hybrid signatures is: "How much extra gas does this cost?" Post-quantum signatures are mathematically more complex than ECDSA, and the additional data they carry must be stored on-chain (at least temporarily). Understanding this cost structure is essential for building economically viable dApps on MST during and after the quantum transition.

💡 Why PQ Signatures Cost More Gas

An ECDSA signature is 65 bytes. A CRYSTALS-Dilithium signature is approximately 2,420 bytes - roughly 37 times larger. Every byte of calldata on an EVM-compatible chain costs gas. Additionally, the on-chain verification of a Dilithium signature involves lattice arithmetic, which consumes more computational gas than the elliptic-curve operations used for ECDSA. The combination of larger calldata and heavier computation means that a hybrid transaction naturally incurs a higher gas cost than a purely classical one.

Cost Component Classical (ECDSA Only) Hybrid (ECDSA + Dilithium) Overhead
Signature calldata 65 bytes (~1,040 gas) 65 + 2,420 bytes (~39,760 gas) +38,720 gas
Signature verification (compute) ~3,000 gas (ecrecover) ~3,000 + ~23,000 gas (Dilithium precompile) +23,000 gas
Key registry lookup (SLOAD) N/A ~2,100 gas (cold) / ~100 gas (warm) +2,100 gas (first call)
Total for a simple transfer ~21,000 gas ~84,820 gas ~4x classical cost

At first glance, a 4x overhead sounds prohibitive. However, three design decisions in MST's architecture absorb this cost and make hybrid transactions practical for everyday use.

1
Dilithium Verification Precompile
MST includes a native precompiled contract (at address 0x0A) for Dilithium signature verification, similar to how Ethereum provides ecrecover at 0x01. Precompiles execute native code rather than EVM bytecode, reducing the compute cost from an estimated ~120,000 gas (if implemented in Solidity) to approximately 23,000 gas. This single optimisation cuts the total overhead by more than 60%.
2
MST's Sub-Paisa Gas Pricing
Because MST targets a gas price of approximately 1-5 gwei and the MST token's unit value is designed for micro-transactions, even 84,820 gas translates to a fraction of a paisa per transaction. Consider that a typical Ethereum transfer at 30 gwei costs roughly $0.50-$2.00. On MST, the same hybrid transaction would cost approximately ₹0.004 (less than one-hundredth of a paisa). For BharatFi's lending protocol processing 50,000 transactions per day, the quantum security overhead adds less than ₹200 per day - a negligible cost for protecting ₹500 crore in TVL.
3
Post-Transition Optimisation
After the ECDSA-drop phase (Ht + 6 months), transactions carry only the Dilithium signature. This removes the 65-byte ECDSA calldata and the ecrecover computation, bringing the total cost back down to roughly 63,000 gas - only 3x the classical cost. Future EIP-style proposals within MST governance aim to reduce this further through signature aggregation, where a single Dilithium proof covers an entire batch of transactions.
🗝️ The Real Cost of NOT Going Quantum-Safe

A 4x gas overhead is a rounding error compared to the cost of a quantum break. If ECDSA is broken and a protocol has not migrated, the entire TVL is at risk. For BharatFi, that means ₹500 crore of user funds stolen in a single afternoon. The cost-benefit analysis overwhelmingly favours paying a fraction of a paisa more per transaction today to avoid catastrophic loss tomorrow. Think of it like buying insurance - the premium is tiny relative to the coverage.

06   SARAL Wallet Quantum Migration

How Your Wallet Silently Becomes Quantum-Safe

From the user's perspective, nothing changes. But under the hood, the wallet performs a choreographed key rotation that would take a manual user hours of terrifying copy-pasting.

1
Onboarding (already quantum-ready)
When a new user creates a SARAL wallet via Google/Facebook login, the secure enclave generates two keypairs: an ECDSA pair for immediate use, and a CRYSTALS-Dilithium pair for future use. The Dilithium public key is immediately registered on-chain via a meta-transaction (gas sponsored by MST's onboarding fund). The user never sees either private key.
2
Receiving the "Transition Notice"
Six months before the transition height, SARAL displays a non-intrusive banner: "MST is upgrading to quantum-safe signatures. Your wallet is already ready." No action required. In the background, the wallet begins signing transactions with both keys immediately, even before the transition height, to test hybrid propagation on mainnet.
3
At Transition Height
The network begins enforcing hybrid signatures. SARAL wallets automatically append the Dilithium signature to every transaction. Legacy wallets that haven't been upgraded stop working - but users can recover their funds through the force-migrate relay using their original SARAL biometric authentication. The migration is complete without a single seed phrase written down.

07   Edge Cases & Failure Modes

What Could Go Wrong - and How MST Handles It

No migration of this scale is without risk. Responsible engineering requires identifying every plausible failure mode in advance and designing explicit recovery procedures. Below, we examine the most critical edge cases in MST's quantum migration, the safeguards built into the protocol, and the rollback strategies available to validators and governance.

1
Google/Facebook Account Compromise During Migration
SARAL wallets use social login as an authentication layer, but private keys live in the device's secure enclave - not in Google's or Facebook's servers. If a user's Google account is compromised, the attacker gains access to the social login token but not to the enclave-stored keys. The attacker could attempt to trigger a key rotation from a new device, but SARAL requires biometric confirmation (fingerprint or face scan) on the original device before any key change is authorised. Additionally, a 48-hour time-lock on key registry updates gives the legitimate user time to detect and cancel the rotation. In the worst case, the user can invoke the emergencyFreeze() function on their identity contract, which blocks all outgoing transactions until the dispute is resolved through MST's on-chain arbitration module.
2
Bug in the Key Registry Contract
The key registry contract is a critical piece of infrastructure - if it returns incorrect public keys, legitimate transactions will be rejected and fraudulent ones may be accepted. To mitigate this risk, MST deploys the registry behind a transparent proxy pattern with a governance-controlled upgrade mechanism. The contract has been formally verified using the Certora prover, covering properties such as "a registered key can only be changed by the address that registered it" and "getPQKey never returns a key for an address that has not called register." In the event a critical bug is discovered post-deployment, governance can pause the contract (halting new registrations while existing ones remain readable) and deploy a patched implementation within hours. A secondary read-only mirror contract stores a snapshot of all registrations as a fallback data source for validators.
3
Validator Fails to Register PQ Key Before Transition
If a validator misses the registration deadline, they are excluded from the active validator set at the transition height. Their staked tokens are not slashed - only frozen. They have a 30-day grace period to register a valid Falcon public key and rejoin. During this grace period, their delegators' rewards continue to accrue in escrow. If the validator fails to register within 30 days, delegators can redelegate to an active validator without incurring an unbonding penalty. This design prevents delegators from being punished for their validator's negligence.
4
Rollback Procedures
In the unlikely event that the transition causes a systemic failure (for example, a bug in the Dilithium precompile that causes all hybrid transactions to fail), MST's governance can execute an emergency rollback to the last pre-transition block. The rollback procedure works as follows: (a) a supermajority of validators (67%+) sign a rollback proposal referencing the safe block height, (b) the network halts at the current height, (c) all nodes revert to the safe height and resume with pre-transition rules, (d) the transition height is rescheduled after the bug is patched. This procedure has been rehearsed on testnet four times, with an average recovery time of 12 minutes. The key constraint is that rollbacks are only possible within the first 72 hours after the transition height - after that, the volume of post-transition state changes makes reversal impractical.
🗝️ Defence in Depth

Notice how each failure mode has multiple layers of protection: biometric locks, time-locks, formal verification, governance pauses, grace periods, and rehearsed rollbacks. This defence-in-depth approach is borrowed from aerospace engineering, where no single failure should bring down the system. In blockchain terms, it means that even if one safety mechanism fails, at least two others stand between the user and loss of funds.

08   Validator Consensus Upgrade

Trika POSA Validators: From ECDSA to Falcon

Securing user transactions is only half the battle. The consensus layer - where validators propose and finalise blocks - must also become quantum-resistant. If a quantum adversary can forge a validator's block signature, they can finalise malicious blocks, double-spend, or halt the chain.

1
Validator Key Registration (Pre-Transition)
Six months before the transition height, validators must submit a Falcon public key (or Dilithium, depending on governance) to a dedicated validator registry contract. They continue signing blocks with ECDSA. The network enters a "shadow-signing" phase: validators optionally append a Falcon signature to block proposals, and other validators verify it in addition to ECDSA. This tests the new signature scheme under live load without enforcement.
2
Dual-Signature Consensus (Transition Height)
At the transition height, the consensus rules change: every block proposal must carry a valid Falcon signature from the proposer, in addition to the ECDSA signature. Validators that have not registered a pq key are excluded from the active set. The fork-choice rule remains unchanged - the chain with the most stake-weighted votes wins - but votes themselves are now hybrid-signed.
3
Post-Transition (ECDSA Dropped)
After a second governance-defined height (typically 6 months after the first), validators stop requiring ECDSA signatures on blocks. The block size shrinks, throughput increases slightly, and the consensus layer is fully post-quantum. Any validator still relying on ECDSA is slashed for failing to produce valid blocks.
💡 Why Falcon for Validators?

Falcon signatures are ~700 bytes vs Dilithium's ~2.5 KB. Validators sign hundreds of blocks per epoch, so bandwidth and verification speed matter. Falcon's smaller signatures reduce block propagation time, keeping MST's 3-second block time intact. Dilithium is used for user transactions because its simpler signing algorithm is more suitable for mobile wallets and secure enclaves with limited computational power.

12   Developer Preparation Guide

What Developers Should Do TODAY to Quantum-Proof Their Smart Contracts

The transition height has not arrived yet, but that does not mean developers should wait. Contracts deployed today will still be running when quantum computers become a threat. If your contract's access control relies solely on msg.sender (which is derived from an ECDSA signature), it is vulnerable the moment ECDSA breaks. The following steps ensure your contracts are quantum-ready from deployment day.

1
Import and Use the IPQKeyRegistry Interface
MST provides a standardised interface at a well-known address. Import it in your contracts and store a reference to the registry. This costs nothing at deployment time and gives your contract the ability to verify that callers have registered post-quantum keys. The interface is available in MST's npm package: @mst/contracts/interfaces/IPQKeyRegistry.sol.
2
Write Quantum-Safe Access Control Modifiers
Create a reusable modifier that checks whether the caller has a registered PQ key. Apply this modifier to any function that controls high-value operations: withdrawals, ownership transfers, governance votes, parameter changes. The modifier does not enforce hybrid signing at the protocol level (that is the validator's job), but it ensures that only quantum-ready accounts can interact with your contract's sensitive functions. This is a defence-in-depth measure - even if a quantum attacker forges an ECDSA signature, your contract independently verifies PQ registration.
3
Test with Hybrid Transactions on Testnet
MST's public testnet (Lakshya) already supports hybrid transaction submission. Configure your Hardhat or Foundry project to use the MST testnet RPC endpoint and enable the hybrid transaction plugin. Deploy your contracts, call your PQ-enforced functions with a test wallet that has a registered Dilithium key, and verify that the modifier works correctly. Also test the negative case: call the same function from a wallet without a PQ key and confirm that the transaction reverts with "No PQ key registered."
4
Plan for Key Rotation in Long-Lived Contracts
Post-quantum cryptography is an evolving field. The algorithm that is secure today (Dilithium) may need to be replaced by a stronger scheme in the future. Design your contracts to read the PQ key from the registry at call-time rather than storing a copy at deployment. This way, when a user rotates their PQ key (for example, upgrading from Dilithium to a future NIST-approved scheme), your contract automatically uses the new key without redeployment. Avoid hard-coding any cryptographic assumptions into your contract logic.
💻 Complete Quantum-Safe Modifier Pattern
import "@mst/contracts/interfaces/IPQKeyRegistry.sol";

abstract contract QuantumSafeBase {
    IPQKeyRegistry public immutable pqRegistry;

    constructor(address _registry) {
        pqRegistry = IPQKeyRegistry(_registry);
    }

    modifier requireQuantumReady(address account) {
        require(
            pqRegistry.getPQKey(account).length > 0,
            "Account has no registered PQ key"
        );
        _;
    }
}

contract SecureTreasury is QuantumSafeBase {
    mapping(address => uint256) public balances;

    constructor(address registry) QuantumSafeBase(registry) {}

    function deposit() external payable {
        balances[msg.sender] += msg.value;
    }

    function withdraw(uint256 amount) external requireQuantumReady(msg.sender) {
        require(balances[msg.sender] >= amount, "Insufficient balance");
        balances[msg.sender] -= amount;
        payable(msg.sender).transfer(amount);
    }
}
Best Practice: Even if you are building a simple ERC-20 token, add the requireQuantumReady modifier to the transfer and approve functions. This protects token holders from quantum-forged approvals that could drain their balances.

🛠️ Smart Contract with Quantum-Enforced Access
contract TimeLockedVault {
    IPQKeyRegistry public pqRegistry;
    address public owner;
    uint256 public unlockTime;

    modifier onlyOwnerQuantumSafe() {
        require(msg.sender == owner, "Not owner");
        require(pqRegistry.getPQKey(msg.sender).length > 0, "Owner must register PQ key");
        _;
    }

    function withdraw() external onlyOwnerQuantumSafe {
        require(block.timestamp >= unlockTime, "Too early");
        payable(owner).transfer(address(this).balance);
    }
}

The onlyOwnerQuantumSafe modifier ensures that even if the owner's ECDSA key is compromised (quantum attack), an attacker cannot call withdraw because they would also need to forge a valid Dilithium signature - which remains quantum-hard. By deploying with this pattern, BharatFi's vaults are immune to the Y2Q break.

Action item: In your capstone project, add a quantum-safe modifier to any contract that holds long-term assets. Use MST's IPQKeyRegistry interface to check that the caller has registered a post-quantum key.

Key Concepts - Know These Cold

01
Hybrid transactions carry both ECDSA and PQC sigs
After transition height, validators require both. Before that, the PQC sig is optional. This enables backward compatibility.
02
Key Registry Contract binds PQ key to address
No new address format. Validators look up the PQ key on-chain during verification. Users never change addresses.
03
SARAL generates Dilithium keypair at onboarding
The post-quantum key is created silently and registered via meta-transaction, making the wallet quantum-ready from day one.
04
Validator upgrade uses Falcon for smaller signatures
Validators register Falcon keys, shadow-sign before transition, and are slashed if they don't comply after transition height.
05
Force-migrate relay protects stranded assets
Users who haven't registered a PQ key can have their assets moved to a secure post-quantum address by a permissionless relay contract.
06
Governance controls transition timing
The height is set by on-chain vote, allowing the community to react to quantum breakthroughs without protocol hard forks.

Key Terms - Sub-Module 17.2

Hybrid Transaction
A transaction containing both an ECDSA signature and a post-quantum signature, enforced during MST's quantum migration window.
Key Registry Contract
An on-chain contract that stores the mapping from an MST address to its registered post-quantum public key.
Transition Height (Hₜ)
The block number at which validators begin enforcing hybrid signatures. Set by on-chain governance.
Shadow-Signing
A pre-transition phase where validators optionally append post-quantum signatures to test infrastructure without enforcement.
Force-Migrate Relay
A permissionless contract that moves assets from a non-upgraded user's address to a quantum-safe address, using pre-authorised allowances.
Dilithium
A NIST-standardised lattice-based digital signature scheme used for user transactions in MST's post-quantum system.
Falcon
A lattice-based signature scheme with compact signatures, used by MST validators for block signing due to its efficiency.
Soft Fork
A protocol upgrade where new rules are a subset of old rules; non-upgraded nodes still accept the new blocks, avoiding a chain split.
Y2Q (Years to Quantum)
The estimated number of years until a cryptographically relevant quantum computer exists - the point at which current elliptic-curve signatures become forgeable. Analogous to "Y2K" but for cryptography.
Algorithm Agility
The ability of a protocol to swap one cryptographic algorithm for another without requiring a hard fork or breaking backward compatibility. MST achieves this through its governance-controlled key registry pattern.
Meta-Transaction
A transaction where the gas fee is paid by a third party (relayer) rather than the user. MST uses meta-transactions during SARAL onboarding so new users can register their PQ keys without holding any tokens.
Shadow Mode
An operational phase where new protocol features run alongside existing ones for testing purposes, without enforcement. Validators in shadow mode produce optional PQ signatures that are verified but not required for block acceptance.
Dual-Stack Format
A transaction encoding that carries two independent signature payloads (classical and post-quantum) in a single serialised structure, allowing validators to verify both without modifying the transaction hash.
Key Rotation
The process of replacing an existing cryptographic key with a new one. In MST, key rotation is performed through the key registry contract and can upgrade a user from one PQC algorithm to a newer one as standards evolve.
