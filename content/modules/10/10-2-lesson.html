02   Real-World Story

The Protocol That Couldn't Fix a Critical Bug

📖 Story

In 2020, a DeFi lending protocol launched with over $50 million locked in its smart contracts. A month after launch, a security researcher discovered a vulnerability in the interest rate calculation that could allow borrowers to drain the lending pool. The team had no upgrade mechanism. The contract was immutable. Their only option was to pause the contract, deploy a new one, and ask users to manually migrate their funds - a process that took weeks, cost millions in gas, and shattered user trust. Several users didn't migrate, and their funds remained locked in the vulnerable contract forever.


Compare this to Compound, one of the largest DeFi protocols. Compound uses a proxy pattern that allows the governance to upgrade the implementation logic while keeping the same contract address. When bugs are discovered, a new implementation is deployed, and the proxy's pointer is updated with a governance vote. Users don't need to move their funds. The contract address stays the same. The upgrade is seamless. This is not a luxury - in blockchain, where immutability is the default, the ability to fix bugs is a critical safety net.


In this sub-module, you will learn the design patterns that give you the best of both worlds: the trustlessness of blockchain with the flexibility to fix bugs and add features. These patterns are not academic - they are running in production, securing billions of dollars in value, right now.

03   The Factory Pattern - Deploying Contracts Like a Production Line

Why Deploy One When You Can Deploy Thousands?

A Factory contract is a contract whose primary job is to deploy other contracts. It's a factory in the literal sense: you call a function, and it produces a brand-new contract instance. This pattern is used everywhere in Web3: Uniswap's factory deploys trading pairs, OpenSea's factory deploys NFT collections, and DAO factories deploy governance contracts.

The problem with deploying directly: Deploying a contract from an EOA (externally owned account) costs gas and requires the deployer to hold the full bytecode. If you want to deploy 100 instances of the same contract, you would pay 100 times the deployment cost. Worse, you would need to send 100 separate transactions.

The factory solution: The factory contract holds the bytecode (or references it) and deploys new instances with a single function call. This is cheaper because the factory can use optimized deployment techniques, and it can batch multiple deployments in one transaction.

// Basic Factory Contract
contract TokenFactory {
    address[] public deployedTokens;

    function deployToken(string memory name, string memory symbol) public returns (address) {
        Token newToken = new Token(name, symbol, msg.sender);
        deployedTokens.push(address(newToken));
        return address(newToken);
    }
}

contract Token {
    string public name;
    string public symbol;
    address public owner;

    constructor(string memory _name, string memory _symbol, address _owner) {
        name = _name;
        symbol = _symbol;
        owner = _owner;
    }
}

The new keyword in Solidity deploys a fresh contract. The factory pattern simply wraps this in a reusable function. But this still deploys the full bytecode each time. For cases where you need thousands of identical contracts (like Gnosis Safe wallets or Uniswap pairs), there's a much cheaper approach: the Minimal Proxy.

🏭 Indian Context: The Chai Franchise Model

Think of a chai franchise like Chai Point. The headquarters (factory contract) defines the recipes, branding, and operations manual (implementation). Each new outlet (deployed contract) follows the same model but has its own location, staff, and local inventory (storage). Instead of rebuilding the entire operations manual from scratch for each outlet (full deployment), the franchise simply opens a new counter that references the central manual (minimal proxy). The cost per outlet drops dramatically, and if the recipe changes, headquarters updates once and all outlets follow. This is exactly how factory + proxy patterns work in Solidity.

Factory pattern with events and registry: In production, factory contracts typically emit events when new contracts are deployed and maintain a registry of all deployed instances. This makes it easy for frontends and indexers (like subgraphs) to discover all contracts created by the factory.

contract TokenFactory {
    address[] public deployedTokens;
    mapping(address => address[]) public userTokens; // creator -> tokens they deployed

    event TokenDeployed(address indexed creator, address tokenAddress, string name);

    function deployToken(string memory name, string memory symbol) public returns (address) {
        Token newToken = new Token(name, symbol, msg.sender);
        address tokenAddr = address(newToken);
        deployedTokens.push(tokenAddr);
        userTokens[msg.sender].push(tokenAddr);
        emit TokenDeployed(msg.sender, tokenAddr, name);
        return tokenAddr;
    }

    function getDeployedCount() public view returns (uint256) {
        return deployedTokens.length;
    }
}

04   The Minimal Proxy (EIP-1167) - Deploy for 1/10th the Gas

Cloning Contracts Without the Overhead

The Minimal Proxy (also called EIP-1167 or "clone factory") is a tiny contract that delegates all calls to a pre-deployed implementation contract. The proxy itself contains almost no code - just a few bytes that forward every call using delegatecall. Deploying this tiny proxy costs a fraction of the gas of deploying the full implementation.

How it works: Instead of deploying new Token() (which copies all bytecode), the factory deploys a minimal proxy - a contract that stores the implementation address in its bytecode and forwards all calls to it. The proxy is typically only ~50 bytes, vs. thousands of bytes for a full implementation. Gas cost: ~50,000 for the proxy vs. ~500,000+ for a full deployment.

// Using OpenZeppelin Clones library
import "@openzeppelin/contracts/proxy/Clones.sol";

contract TokenFactory {
    using Clones for address;
    address public implementation;

    constructor(address _implementation) {
        implementation = _implementation;
    }

    function cloneToken(string memory name, string memory symbol) public returns (address) {
        address clone = implementation.clone();
        Token(clone).initialize(name, symbol, msg.sender);
        return clone;
    }
}

Important: Minimal proxies rely on delegatecall, which means the implementation contract's code runs in the context of the proxy's storage. This is why the initialization function is separate from the constructor - constructors run in the context of the implementation, not the proxy, so they cannot initialize proxy storage.

CREATE vs CREATE2: CREATE deploys a contract at an address determined by the deployer's address and nonce. CREATE2 deploys at an address determined by the deployer's address, a salt, and the init code hash. CREATE2 enables deterministic addresses - you can predict the address before deployment, which is critical for counterfactual systems like Uniswap V3 and Optimistic Rollups.

05   The Proxy Pattern - Making Contracts Upgradeable

Separating Logic from Storage So You Can Fix Bugs Forever

The core problem: smart contracts are immutable. Once deployed, their code cannot change. But software always has bugs, and requirements change. The Proxy pattern solves this by splitting the contract into two parts: a proxy that holds the storage and the user's address, and an implementation that holds the logic. The proxy delegates all function calls to the implementation using delegatecall.

delegatecall is the magic. When Contract A calls Contract B using delegatecall, Contract B's code runs, but it reads and writes Contract A's storage. It's as if Contract B's code was copy-pasted into Contract A. This is exactly what the proxy does: it delegates to the implementation, which runs the logic and modifies the proxy's storage.

FIGURE 10.2-A · Proxy Pattern Architecture
User → Proxy Contract → Implementation V1
         (storage)        (logic)

When an upgrade happens:
User → Proxy Contract → Implementation V2
         (same storage)   (new logic, same storage layout)
The proxy holds the storage and delegates calls. The implementation can be swapped without changing the proxy's address or state.

Storage collisions - the biggest danger: When you upgrade to a new implementation, the new implementation must use the exact same storage layout as the old one. If you add a new state variable at the beginning, it will overwrite the old first variable, corrupting your data. The solution is to use unstructured storage - storing critical data at pseudo-random slots derived from hashes, so they never collide with normal Solidity storage.

// Unstructured storage pattern (OpenZeppelin)
bytes32 private constant IMPLEMENTATION_SLOT = 
    keccak256("eip1967.proxy.implementation") - 1;

function _setImplementation(address newImpl) internal {
    assembly { sstore(IMPLEMENTATION_SLOT, newImpl) }
}

function _getImplementation() internal view returns (address impl) {
    assembly { impl := sload(IMPLEMENTATION_SLOT) }
}

This pattern stores the implementation address at a slot that is astronomically unlikely to collide with any Solidity-assigned slot, preventing overwrites.

Transparent Proxy vs. UUPS: There are two main upgradeable proxy patterns. Transparent Proxy (EIP-1967) has the upgrade logic in the proxy itself. Only an admin can upgrade. UUPS (Universal Upgradeable Proxy Standard) puts the upgrade logic in the implementation, making the proxy simpler and cheaper, but requiring all implementations to include upgrade functionality. OpenZeppelin supports both.

⚠️ The UUPS Upgrade Trap

With UUPS, the upgrade function lives in the implementation. If you deploy a new implementation that accidentally removes or breaks the upgradeTo() function, the proxy becomes permanently stuck on that version - there is no way to upgrade again. This is called "bricking" the proxy. Always include comprehensive tests that verify the upgrade path before deploying any new implementation. OpenZeppelin's UUPSUpgradeable base contract handles this automatically, but if you write a custom implementation, this is a critical risk.

Step-by-step: Deploying an upgradeable contract with OpenZeppelin:

Step 1: Write your implementation contract. Inherit from Initializable and UUPSUpgradeable. Replace the constructor with an initialize function marked with the initializer modifier.

Step 2: Deploy the implementation contract normally.

Step 3: Deploy a proxy (using OpenZeppelin's ERC1967Proxy) that points to your implementation and passes the initialize call data.

Step 4: Users interact with the proxy address. To upgrade, deploy a new implementation and call upgradeTo(newAddress) on the proxy (via the old implementation's upgrade function).

Step 5: After upgrading, the proxy now delegates to the new implementation. All storage is preserved because the proxy's storage is untouched - only the code pointer changed.

🔑 The Golden Rule of Upgrades

Never change the order of existing state variables in your implementation. You can add new variables at the end, but you must never remove, reorder, or change the type of existing variables. The proxy's storage slots are assigned based on declaration order - changing the order corrupts all stored data. OpenZeppelin recommends using a __gap array (a reserved uint256[50] at the end of each contract) to leave room for future variables without breaking the storage layout.

06   The Diamond Standard (EIP-2535) - Unlimited Contract Size

When Your Contract Needs 100 Functions Across 10 Modules

Ethereum imposes a contract size limit of 24,576 bytes. For complex protocols, a single implementation contract may hit this limit. The Diamond Standard solves this by allowing a single proxy (called a diamond) to have multiple implementation contracts (called facets). Each facet provides a subset of functions, and the diamond routes calls to the correct facet.

How it works: The diamond stores a mapping of function selectors (the first 4 bytes of a function signature hash) to facet addresses. When a call comes in, the diamond looks up the selector, finds the corresponding facet, and delegatecalls it. This is managed by a special facet called the DiamondLoupe, which provides introspection functions (list all facets, list their selectors).

// Simplified Diamond Storage
struct Facet {
    address facetAddress;
    bytes4[] functionSelectors;
}

mapping(bytes4 => address) public selectorToFacet;

function diamondCut(Facet[] memory facets) external onlyOwner {
    for (uint i = 0; i < facets.length; i++) {
        for (uint j = 0; j < facets[i].functionSelectors.length; j++) {
            selectorToFacet[facets[i].functionSelectors[j]] = facets[i].facetAddress;
        }
    }
}

fallback() external payable {
    address facet = selectorToFacet[msg.sig];
    require(facet != address(0), "Function not found");
    assembly {
        calldatacopy(0, 0, calldatasize())
        let result := delegatecall(gas(), facet, 0, calldatasize(), 0, 0)
        returndatacopy(0, 0, returndatasize())
        switch result case 0 { revert(0, returndatasize()) } default { return(0, returndatasize()) }
    }
}

The Diamond Standard is the most flexible upgradeability pattern: you can add, replace, or remove individual functions without touching other facets. It's used by protocols like Aavegotchi and Defi Kingdoms, where the contract ecosystem is large and constantly evolving.

Diamond storage pattern: Since all facets share the proxy's storage, you must be careful to avoid storage collisions between facets. The Diamond Standard uses a technique called Diamond Storage: each facet stores its data in a struct located at a unique, hash-derived storage slot. This ensures that facet A's variables never collide with facet B's variables, even if both have a variable named owner.

// Diamond Storage pattern for a Staking facet
library StakingStorage {
    bytes32 constant POSITION = keccak256("diamond.staking.storage");

    struct Layout {
        mapping(address => uint256) stakes;
        uint256 totalStaked;
        uint256 rewardRate;
    }

    function layout() internal pure returns (Layout storage l) {
        bytes32 position = POSITION;
        assembly { l.slot := position }
    }
}

// Used inside the Staking facet:
function stake(uint256 amount) external {
    StakingStorage.Layout storage s = StakingStorage.layout();
    s.stakes[msg.sender] += amount;
    s.totalStaked += amount;
}

07   Putting It All Together - A Complete Example

A Factory That Deploys Upgradeable Tokens

Below is a factory that combines the minimal proxy with upgradeability. It deploys a token as a transparent proxy, allowing the token owner to later upgrade the token logic.

contract UpgradeableTokenFactory {
    address public logicImplementation;

    constructor(address _logic) {
        logicImplementation = _logic;
    }

    function deployToken(string memory name, string memory symbol) public returns (address) {
        // Deploy minimal proxy pointing to logicImplementation
        bytes20 targetBytes = bytes20(logicImplementation);
        address proxy;
        assembly {
            let clone := mload(0x40)
            mstore(clone, 0x3d602d80600a3d3981f3363d3d373d3d3d363d73000000000000000000000000)
            mstore(add(clone, 0x14), targetBytes)
            mstore(add(clone, 0x28), 0x5af43d82803e903d91602b57fd5bf30000000000000000000000000000000000)
            proxy := create(0, clone, 0x37)
        }
        Token(proxy).initialize(name, symbol, msg.sender);
        return proxy;
    }
}

This pattern is used by major protocols to let users deploy their own upgradeable tokens at a fraction of the cost of deploying from scratch. The factory holds a single logic implementation, and every deployed token is a cheap proxy.

09   Glossary

Key Terms - Sub-Module 10.2

Factory Pattern
A contract that deploys other contracts, often using minimal proxies to save gas. Centralises deployment logic and enables deterministic addresses via CREATE2.
Minimal Proxy (EIP-1167)
A lightweight contract that delegates all calls to a pre-deployed implementation. Contains ~50 bytes of bytecode, costing far less than a full deployment.
delegatecall
An EVM opcode that executes another contract's code in the caller's storage context. The key mechanism behind proxy patterns and upgradeable contracts.
Transparent Proxy (EIP-1967)
An upgradeable proxy where the upgrade function lives in the proxy itself. The admin cannot accidentally call implementation functions.
UUPS
Universal Upgradeable Proxy Standard - an upgradeable proxy where the upgrade logic is in the implementation. Cheaper to deploy than Transparent Proxy.
Diamond Standard (EIP-2535)
A multi-facet proxy that routes function calls to different implementation contracts based on function selectors. Solves the contract size limit.
Unstructured Storage
A pattern that stores proxy data at pseudo-random storage slots (derived from hashes) to prevent collisions with implementation storage.
CREATE2
An EVM opcode that deploys a contract at a deterministic address based on the deployer, a salt, and the init code. Enables counterfactual contracts.
Storage Collision
A critical bug where two variables in a proxy system map to the same storage slot, causing one to overwrite the other. Prevented by using unstructured storage and preserving variable order across upgrades.
Initializer
A function that replaces the constructor in upgradeable contracts. Marked with OpenZeppelin's initializer modifier to ensure it can only be called once, preventing re-initialization attacks.
__gap Array
A reserved storage array (typically uint256[50]) placed at the end of an upgradeable contract to reserve storage slots for future state variables without breaking the existing layout.
Diamond Storage
A storage isolation pattern where each facet stores its data in a struct at a unique, hash-derived storage slot, preventing collisions between facets in a Diamond proxy.
Fallback Function
A special Solidity function that executes when a contract receives a call with an unrecognised function selector. In proxy contracts, the fallback forwards all calls to the implementation via delegatecall.
