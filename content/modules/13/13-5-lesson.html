02   Real-World Introduction Story

The Farmer, the Insurance, and the Blockchain Rain Gauge

📖 Story

In the arid fields of Marathwada, a farmer named Tukaram bought a blockchain-based crop insurance policy. The policy promised a payout of ₹50,000 if the monsoon rainfall in his district fell below 400 mm by the end of September. The smart contract held the insurer's funds in escrow. The insurer had deployed a weather oracle that, every day, fetched official rainfall data from the Indian Meteorological Department's API, ran a WASM module to compute the cumulative rainfall, and submitted a ZK proof to the contract. The farmer didn't have to trust the insurer, the API, or even the oracle operator - the ZK proof guaranteed the computation was correct. On October 1st, the contract verified the final proof: rainfall was 387 mm. The payout of ₹50,000 was released to Tukaram's wallet within seconds. The gas cost for the entire oracle over three months? Less than ₹10 on MST.

03   Background & History

Weather Oracles - The First Killer App for Trustless Data

Weather data has been one of the earliest and most compelling use cases for blockchain oracles. Agriculture, insurance, logistics, and energy trading all depend on accurate, tamper-proof weather information. Traditional parametric insurance (payouts triggered by a predefined index, like rainfall amount) has existed for decades, but it relies on a central authority to certify the data. Blockchain changed that by enabling smart contracts to execute payouts automatically - but only if they could trust the incoming weather data.

Early solutions used centralised oracles, but these introduced a single point of failure. Decentralised oracle networks like Chainlink added redundancy, yet they still required multiple trusted nodes and exposed raw data. WASMify, with its ZK-proof approach, allowed a single execution of a complex rainfall computation to be verified on-chain without any trust in the executor. This project simulates exactly that: you'll build a mini-oracle that reads a simulated temperature file, checks if the average is below a threshold, and settles a contract - all trustlessly.

Why this project matters: It's a perfect synthesis of blockchain, zero-knowledge, and real-world data. Completing it demonstrates you can design, code, and deploy a working oracle - a skill that top DeFi and insurance protocols actively seek.

The evolution of weather oracles: Weather oracle technology has progressed through distinct generations, each solving a different trust problem:

1
Generation 1: Centralised API Oracles (2017-2019)
A single server fetched weather data from an API (like OpenWeatherMap) and pushed it on-chain. The obvious problem: the operator could lie, manipulate data, or go offline. If the server went down during a critical monsoon measurement, the insurance contract had no data and no payout could be triggered.
2
Generation 2: Decentralised Oracle Networks (2020-2022)
Chainlink and Band Protocol introduced networks of independent nodes that each fetched the same data and submitted it on-chain. The contract aggregated submissions (e.g., median of 7 nodes). This solved the single-point-of-failure problem but introduced new ones: high cost (7 nodes all paying gas), data exposure (all nodes see the raw API response), and a reliance on the honesty of a majority of nodes.
3
Generation 3: ZK-Proof Oracles (2023-present)
WASMify and similar systems allow a single executor to fetch data, run a computation, and generate a zero-knowledge proof that the computation was performed correctly. The smart contract verifies the proof without needing to trust the executor, see the raw data, or pay for multiple nodes. This is what you are building in this project.
🔑 Why India Needs Weather Oracles

India's agriculture sector employs over 42% of the workforce and contributes 18% of GDP, yet crop insurance penetration remains below 30%. The Pradhan Mantri Fasal Bima Yojana (PMFBY) scheme aims to cover more farmers, but claim settlement is slow because it relies on manual crop cutting experiments and subjective loss assessment. A blockchain-based weather oracle that automatically triggers payouts based on verifiable rainfall data could reduce settlement time from months to seconds. The Indian Meteorological Department (IMD) already publishes district-level rainfall data publicly - the missing piece is a trustless bridge from that data to on-chain insurance contracts. That bridge is exactly what WASMify provides.

04   Beginner-Friendly Explanation

The Oracle You're Building - An Automatic Rainfall Judge

🌧️ The Analogy

Imagine you're an umpire who must decide if a cricket match is rained out. But instead of looking at the sky yourself, you have a locked box. The box contains a thermometer and a calculator. The box can be opened, measurements taken, and then it prints a signed note saying "Rainfall was 387 mm - below 400 mm." The signature can't be forged. You never saw the raw data, but you trust the math. That's your WASM module. The signed note is the ZK proof. The umpire is the smart contract. This project teaches you to build that locked box.

You'll create a simple "weather station" that checks if the average temperature of a shipment or a crop season stayed below a critical threshold. The project will simulate the temperature data, prove the average is below the limit, and release escrowed funds on MST.

①
DATA
Where does the temperature data come from?
For this project, you'll use a simulated JSON file containing an array of temperature readings (one per minute for a 48-hour period). In a real deployment, this would come from IoT sensors or a trusted weather API. The WASM module will read this file as input.
②
COMPUTE
What does the WASM module do?
It takes the temperature array, computes the average temperature, and checks if it's below a threshold (e.g., 8°C for a cold-chain vaccine). It returns 1 (true) if the condition is met, or 0 (false). This is a deterministic, provable computation.
③
PROVE
How is the ZK proof generated?
You'll use a simulated ZK prover (or a real one like RISC Zero if your hardware permits). The prover takes the WASM binary and the temperature data, executes the module, and produces a proof and the public output (1 or 0). For this project, we'll provide a simple Node.js script that simulates proof generation by hashing the output - you can replace it with a real prover later.
④
VERIFY
How does the MST contract use the proof?
A Solidity consumer contract calls the WASMify Verifier contract with the proof and the claimed output. If valid, it executes the business logic - for example, releasing an escrowed payment to a farmer or certifying a shipment. The entire verification costs about ₹0.024 on MST.

06   Technical Deep Dive - Implementation

Let's Build the Weather Oracle Together

Step 1: Write the WASM Module in Rust

Create a file weather_check.rs. The function check_average receives a pointer to a slice of temperature values (floating point) and returns 1 if the average is below the threshold (e.g., 8°C).

// weather_check.rs
#[no_mangle]
pub extern "C" fn check_average(temps_ptr: *const f32, len: usize) -> u32 {
    if len == 0 { return 0; }
    let temps = unsafe { std::slice::from_raw_parts(temps_ptr, len) };
    let sum: f32 = temps.iter().sum();
    let avg = sum / len as f32;
    if avg < 8.0 { 1 } else { 0 }
}

Compile to WASM using rustc --target wasm32-unknown-unknown --crate-type cdylib weather_check.rs -o weather_check.wasm. You now have a portable, deterministic module.

Step 2: Prepare Simulated Temperature Data

Create a JSON file temperatures.json with an array of numbers representing temperatures over 48 hours (one reading per minute = 2880 values). Keep all values below 8°C to test the success case, and include one spike above 8°C to test failure.

Step 3: Simulate the ZK Prover (Node.js)

In a real deployment, you'd use RISC Zero or zkWASM. For this project, we'll simulate proof generation by executing the WASM module and creating a fake proof (a hash of the result). You'll later replace this with a real prover.

// prover.js
const { readFileSync } = require('fs');
const { ethers } = require('ethers');

// Simulate running the WASM module (you can use a WASM runtime like wasmer)
const temps = JSON.parse(readFileSync('temperatures.json'));
const avg = temps.reduce((a,b) => a+b, 0) / temps.length;
const result = avg < 8.0 ? 1 : 0;

// Generate a "proof" (in reality, a ZK proof; here we just hash the result)
const proof = ethers.keccak256(ethers.toUtf8Bytes(result.toString()));
console.log("Result:", result);
console.log("Proof:", proof);

Step 4: Write the Solidity Consumer Contract

// WeatherOracle.sol
interface IWASMifyVerifier {
    function verify(bytes calldata proof, uint256 result) external view returns (bool);
}

contract WeatherOracle {
    IWASMifyVerifier public verifier;
    address public insurer;
    address public farmer;
    uint256 public payoutAmount;
    bool public settled;

    event PayoutReleased(address indexed farmer, uint256 amount);

    constructor(address _verifier, address _farmer) payable {
        verifier = IWASMifyVerifier(_verifier);
        insurer = msg.sender;
        farmer = _farmer;
        payoutAmount = msg.value;
    }

    function settle(bytes calldata proof, uint256 result) external {
        require(!settled, "Already settled");
        require(verifier.verify(proof, result), "Invalid proof");
        settled = true;
        if (result == 1) {
            payable(farmer).transfer(payoutAmount);
            emit PayoutReleased(farmer, payoutAmount);
        }
        // else, no payout
    }
}

Step 5: Deploy to MST Testnet

Use Hardhat. Configure hardhat.config.js with MST Testnet RPC and your private key. Deploy the WeatherOracle with the address of the WASMify Verifier (use a placeholder or a mock for now). Fund the contract with test MSTC.

npx hardhat run scripts/deploy.js --network mst_testnet

Step 6: Submit the Proof from the Off-Chain Script

Modify prover.js to include ethers.js and send a transaction to the deployed contract with the proof and result.

const contract = new ethers.Contract(oracleAddress, abi, wallet);
const tx = await contract.settle(proof, result);
await tx.wait();
console.log("Settled! Tx:", tx.hash);

08B   Extended Use Cases Beyond Insurance

Where Else Can a Weather Oracle Be Used?

The weather oracle pattern you are building is not limited to crop insurance. The same architecture - fetch external data, compute a result off-chain, prove it with ZK, settle on-chain - can be applied to dozens of real-world scenarios:

🚚
Cold-Chain Logistics Verification
Pharmaceutical companies must maintain vaccines at 2-8 degrees Celsius during transport. IoT sensors record temperatures every minute. Your WASM module checks if the average temperature stayed within range. If it did, the smart contract releases payment to the logistics provider. If it didn't, the shipment is flagged and payment is withheld. Companies like Zydus Cadila and Serum Institute could use this for transparent, automated compliance.
⚡
Solar Energy Production Certificates
India's National Solar Mission targets 500 GW of renewable energy by 2030. Solar farms need to prove their production to earn Renewable Energy Certificates (RECs). A weather oracle that reads solar irradiance data and proves the expected energy output was met can automate REC issuance on-chain, eliminating manual audits and paper certificates.
🌊
Flood Warning and Relief Disbursement
During monsoon season, districts in Bihar, Assam, and Kerala regularly face flooding. A weather oracle monitoring river water levels (from Central Water Commission data) can trigger automatic relief fund disbursement when water levels cross a critical threshold. The ZK proof ensures the measurement is genuine, preventing fraudulent claims.
🏟️
Event Cancellation Insurance
Outdoor events (IPL matches, festivals, weddings) can be insured against bad weather. If rainfall exceeds 50mm on the event date, the policy pays out. The oracle reads rainfall data from the nearest IMD station, proves the computation, and settles the contract. No claims adjuster needed.

⚠️ Security Consideration: The Oracle Problem Is Not Fully Solved

Even with ZK proofs, your oracle is only as trustworthy as the data source. The ZK proof guarantees that the computation on the input data was correct - it does not guarantee that the input data itself is accurate. If the API you fetch from is compromised or returns incorrect data, the proof will still validate, but the result will be wrong. In production, use multiple independent data sources and implement outlier detection in your WASM module to reject data that is statistically implausible (e.g., a temperature reading of 80 degrees C in Delhi).

09   Common Pitfalls

Debugging Tips for Your Project

⚠️
WASM module returns wrong result
Double-check that you're using the correct data types (f32) and that the array length matches. Test the module locally with a small array before proving.
⚠️
Proof verification fails on-chain
Ensure the proof format matches what the Verifier expects. In the simulated version, your contract's verify function must mimic the prover's logic. In production, use the standard verifier contract.
⚠️
Running out of test MSTC
Use the MST Faucet to get test tokens. Remember that even 0.1 MSTC is enough for thousands of transactions.
11   Glossary

Key Terms - Sub-Module 13.5

WASM Module
A compiled WebAssembly binary that performs a deterministic computation. Used in WASMify as the off-chain logic to be proven.
ZK Prover
Software that generates a zero-knowledge proof of a computation. In this project, simulated with a hash; in production, a real prover like RISC Zero.
Consumer Contract
The on-chain Solidity contract that consumes the verified data and executes business logic (e.g., insurance payout).
Parametric Insurance
An insurance model where payout is automatically triggered by a predefined index (like rainfall) rather than an assessed loss.
Deterministic Computation
A computation that always produces the same output for the same input, regardless of when or where it is executed. Required for ZK-provable WASM modules.
IMD (Indian Meteorological Department)
India's official weather authority that publishes district-level rainfall, temperature, and forecast data. A potential data source for production weather oracles.
Cold-Chain Monitoring
The process of tracking temperature conditions during the transport and storage of temperature-sensitive goods like vaccines and food. A key use case for IoT-connected weather oracles.
Escrow
A smart contract pattern where funds are held by the contract and released only when predefined conditions are met (e.g., proof verification). Used in the WeatherOracle to hold the insurer's payout.
Data Freshness
The requirement that oracle data be recent enough to be relevant. Production oracles include timestamp checks to reject stale data that could be manipulated or irrelevant.
Outlier Detection
A technique in the WASM module that rejects individual data points that are statistically implausible (e.g., 80 degrees C in winter), protecting against corrupted or manipulated input data.
