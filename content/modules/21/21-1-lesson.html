02 · Real-World Introduction Story

The ₹120 Crore Lesson - Why Audits Are Not Optional

📖 Story

In March 2022, a popular DeFi protocol on an EVM chain was preparing for a V2 launch. The team had built a beautiful UI, a clever staking mechanism, and a governance token. The smart contract was written by a senior developer and reviewed by two other engineers. They ran the Hardhat test suite - all green. The code was deployed to mainnet on a Friday evening.


By Monday morning, ₹120 crore (about $15 million) was gone. The attacker had exploited a subtle reentrancy vulnerability in the withdraw() function. The team had missed it because their test only checked normal withdrawal flow - not cross-contract reentrancy. Worse, they had never run a single automated security tool. No Slither. No Mythril. No fuzzing. Just manual confidence.


This is not a rare story. In 2023 alone, blockchain hacks totalled over $1.8 billion, according to Chainalysis. The vast majority of exploited contracts had never undergone a professional audit or even a basic automated scan. The MST ecosystem is safer by design - but EVM compatibility means existing Solidity vulnerabilities can still travel to MST. As an MST Blockchain Developer, your capstone project will handle real user value. You owe it to yourself and your users to harden every line of code.

💡 Indian Context - WazirX Exchange Breach (2024)

Closer to home, the WazirX exchange suffered a security breach in July 2024 that resulted in over ₹2,000 crore ($230 million) being stolen from a multisig wallet. The attackers exploited a mismatch between what the multisig interface showed to signers and what the actual transaction payload contained. This was not a Solidity bug - it was a transaction verification failure at the human layer. The lesson is clear: security is not just about code. It involves every step from contract logic to the signing interface, from access control to operational procedures. Indian developers building on MST must internalise this: the stakes are real, the attackers are sophisticated, and "it works on testnet" is never enough.

🗝️ The Cost of Not Auditing - A Data Snapshot

According to DeFiLlama and Rekt.news data, the top 10 DeFi exploits of 2023-2024 resulted in a combined loss exceeding $3.5 billion. Of these, 7 out of 10 projects had never undergone a formal security audit. Of the 3 that were audited, 2 had modified code after the audit without re-auditing. The single project that was properly audited and still exploited was hit through a vulnerability in an external dependency - a reminder that audit scope must include all imported libraries and oracle integrations. For MST capstone projects, even a self-conducted audit using free tools like Slither dramatically reduces risk compared to no audit at all.

03 · Why Final Hardening Matters on MST

EVM Compatibility = Attack Surface Transportability

Because MST is fully EVM-compatible, any security flaw that exists in Ethereum’s Solidity ecosystem can manifest identically on MST. The low gas fees on MST actually increase the risk - attackers can afford to try multiple exploits that would be prohibitively expensive on Ethereum. Moreover, MST’s Trika POSA provides fast finality (~3 seconds), but that also means a successful attack is irreversible almost instantly.

A robust security audit is not just a checkbox for your capstone project; it’s the difference between a funded startup and a catastrophic failure. MST’s BuildGrant requires an audit report signed by a qualified reviewer (even if self-audited with tools). The skills you learn here will be directly applicable in job interviews, where 80% of technical questions revolve around security and gas optimisation.

Consider the unique risk factors that apply specifically to MST deployments. The low gas fees that make MST attractive for micro-transactions (think RWA fractional ownership, gaming rewards, loyalty token programs) also mean that an attacker can execute thousands of exploit attempts for less than ₹10. On Ethereum, a single complex attack transaction might cost $50-$200 in gas, which naturally deters brute-force attempts. On MST, that same attack costs fractions of a paisa. This economic reality means your contracts must be hardened against not just known vulnerability patterns, but also against high-frequency probing attacks.

Furthermore, MST's growing ecosystem of bridges (connecting to Ethereum, BSC, and Polygon) introduces cross-chain attack vectors. If your capstone interacts with bridged assets, you must validate that the bridge contract on the MST side is the official one. Attackers have historically deployed fake bridge contracts that mimic the interface of legitimate ones but route funds to the attacker's address. Always verify external contract addresses against the official MST documentation before hardcoding them.

04B · The Four-Phase Audit Process

From Raw Code to Mainnet-Ready - A Structured Approach

Professional audit firms like Trail of Bits, OpenZeppelin, and Consensys Diligence follow a structured multi-phase process. Even when self-auditing your capstone project, adopting this framework dramatically improves the quality of your review.

P1
Phase 1 - Scoping and Documentation Review (Day 1)
Before touching a single line of code, the auditor reads all documentation: the README, architecture diagrams, protocol specifications, and tokenomics models. The goal is to understand the intended behaviour of the system. Without this context, an auditor cannot distinguish between a feature and a bug. For your capstone, write a one-page "Security Scope" document that lists every contract, every external dependency, and every privileged role (owner, admin, minter). This document becomes the reference point for the entire audit.
P2
Phase 2 - Automated Scanning (Day 1-2)
Run Slither, Mythril, and Echidna against all contracts. This phase is fast and catches the "low-hanging fruit" - common vulnerabilities like reentrancy, unchecked return values, and missing access modifiers. The key discipline here is to triage every finding, not ignore false positives. Create a spreadsheet with columns: Finding ID, Tool, Severity, Description, Status (Fixed / Acknowledged / False Positive), and Justification. This spreadsheet becomes part of your audit report.
P3
Phase 3 - Manual Code Review (Day 2-4)
The human auditor reads every function, every modifier, and every state variable. They trace the flow of funds through the protocol, looking for economic exploits that tools cannot detect. They ask questions like: "Can a user withdraw more than they deposited?", "Can the owner rug-pull by changing a critical address?", "What happens if a user sends 0 tokens?", "What if block.timestamp is manipulated by 15 seconds?". This phase requires deep Solidity expertise and a suspicious mindset.
P4
Phase 4 - Report Writing and Remediation (Day 4-5)
The final report documents every finding with a clear description, severity rating (Critical / High / Medium / Low / Informational), proof of concept (PoC) code, and a recommended fix. After the developer applies fixes, the auditor performs a re-audit to verify that fixes are correct and haven't introduced new issues. For your capstone, the re-audit is simply re-running all tools and re-reviewing the modified functions.
💡 Time Investment - Self-Audit vs Professional Audit

A professional audit from a top-tier firm costs $15,000-$100,000+ and takes 2-6 weeks. For a capstone project, a self-audit following this four-phase process takes approximately 8-12 hours of focused work. The difference in cost is enormous, but the improvement in security is significant even at the self-audit level. Many MST BuildGrant recipients have successfully secured funding with well-documented self-audits using Slither, Mythril, and manual review. The key is thoroughness and documentation - not the price tag.

05 · Automated Analysis - Your First Line of Defence

Slither, Mythril, and Echidna on MST

1
Slither - Static Analysis
Slither (by Trail of Bits) is the industry-standard static analyser for Solidity. It examines your code without executing it, detecting reentrancy, missing events, incorrect modifiers, unused variables, and dangerous delegatecall patterns. Install via pip3 install slither-analyzer. Run it with slither . inside your Hardhat project. Slither outputs a ranked list of findings with severity: High, Medium, Low, Informational. For MST deployment, fix all High and Medium issues before proceeding.
2
Mythril - Symbolic Execution
Mythril explores all possible execution paths of a contract, looking for assertion violations, integer overflows, and unprotected SELFDESTRUCT calls. Unlike Slither, it doesn’t just read the code - it runs it symbolically, reaching deep states. Command: myth analyze contracts/MyContract.sol --execution-timeout 60. It can uncover bugs that a human might miss in nested if-else branches.
3
Echidna - Fuzzing / Invariant Testing
Echidna bombards your contract with random inputs and checks whether any invariant breaks (e.g., “total supply must never exceed 1 million”). You write invariants as Solidity functions with names starting with echidna_. This catches logic errors that static tools miss. For MST, we especially fuzz staking rewards and RWA token mint/burn functions.

Interpreting Tool Output - A Practical Example

Understanding what these tools report is just as important as running them. Let's look at a realistic Slither output and learn to read it like a professional auditor.

$ slither . ... StakingPool.withdraw(uint256) (contracts/StakingPool.sol#14-18) External calls: - (ok) = msg.sender.call{value: amount}("") (contracts/StakingPool.sol#17) State variables written after the call(s): - stakes[msg.sender] -= amount (contracts/StakingPool.sol#16) ⚠️ Reentrancy in StakingPool.withdraw(uint256) Reference: https://github.com/crytic/slither/wiki/Detector-Documentation#reentrancy INFO:Detectors: StakingPool.updateReward(address,uint256) has no access control Reference: https://github.com/crytic/slither/wiki/Detector-Documentation#missing-access-control 3 result(s) found

Notice how Slither tells you the exact line numbers, the specific state variable written after the external call, and provides a reference link for each detector. A common beginner mistake is to see "3 results found" and assume the contract has three bugs. In reality, Slither may report false positives - especially for patterns where you intentionally use unchecked blocks or where the external call is to a trusted contract. The auditor's job is to evaluate each finding individually and mark it as either "Confirmed" (requires fix), "Acknowledged" (accepted risk with justification), or "False Positive" (not actually exploitable in this context).

⚠️ Common Slither False Positives on MST

Slither may flag MST-specific patterns as issues. For example, if your contract calls the SARAL relayer contract (a trusted MST system contract), Slither will flag it as an "external call to untrusted contract." In this case, you mark it as a false positive and document why: "The SARAL relayer at address 0x... is an MST system contract deployed by the MST Foundation. Its bytecode is verified on MST Explorer." Always document your reasoning - this is what distinguishes a professional audit from a superficial one.

06 · Manual Code-Review Checklist

The Human Touch - 12 Points You Must Verify

Automated tools are essential, but they cannot understand business logic. A contract may have zero tool-reported issues yet contain a fatal economic design flaw. Here is the checklist that professional auditors use for every MST dApp.

01
Reentrancy (Checks-Effects-Interactions)
All state changes must happen before external calls. Use OpenZeppelin’s ReentrancyGuard. Never rely on transfer() or send() for reentrancy protection - use CEI pattern.
02
Integer Overflow / Underflow
Solidity ≥0.8 has built-in checks, but explicit unchecked blocks can be dangerous. Verify every unchecked usage with a detailed comment.
03
Access Control
Every state-changing function must have appropriate modifiers (onlyOwner, role-based). Check for missing onlyOwner on critical setters. Avoid tx.origin for authentication.
04
Front-Running & MEV
On MST, validators can still reorder transactions. Use commit-reveal schemes for sensitive actions, and avoid large price-impact operations that can be sandwiched.
05
Unchecked Return Values
Low-level calls (.call()) return a boolean that must be checked. Failing to check can lead to silent failures. Use require(success, "call failed").
06
Denial of Service (DoS)
Avoid loops over unbounded arrays that can hit block gas limits. Use pull-over-push for payments, and never rely on a single external call to succeed for critical logic.
07
Timestamp Dependence
Block timestamps can be manipulated by validators within a small window (~30 seconds). Do not use them for precise randomness or as a sole source of truth for deadlines (use block numbers where appropriate).
08
Delegatecall & Proxy Security
If you use upgradeable contracts, ensure storage slots never clash between implementation versions. Use OpenZeppelin’s TransparentUpgradeableProxy or UUPS with caution.
09
Signature Replay Attacks
If your contract accepts off-chain signatures, include a nonce and chain ID to prevent cross-chain replay. Use EIP-712 structured hashing.
10
Rounding Errors & Precision
In financial contracts, always multiply before dividing to minimise precision loss. Use fixed-point math libraries (PRBMath) for high-accuracy calculations.
11
Event Emission
All state-changing functions should emit events for off-chain monitoring. Missing events make debugging and user-notification impossible.
12
Business Logic & Invariants
Define invariants explicitly (“token A balance + token B balance in pool must equal constant k”). Write Echidna invariants for these. Review any state transition that could break them.
07 · Gas Optimisation - Every Wei Counts

MST Gas Model and Why Optimisation Still Matters

MST’s gas fees are significantly lower than Ethereum’s during congestion (often < 0.01 MST per transfer). However, gas optimisation is still critical for three reasons:

  • Scalability: high-throughput dApps (gaming, RWA micro-transactions) can execute millions of transactions daily. Shaving 20% off each call saves real money.
  • User experience: lower gas = faster confirmation and less friction. This is part of MST’s promise.
  • MST BuildGrant criteria: grant reviewers penalise wasteful code; a gas-optimised contract signals professionalism.
❌ Common Gas Wasters
  • Using memory instead of calldata for external function parameters
  • Reading storage variables multiple times in a loop
  • Unnecessary public visibility (use external)
  • Storing large arrays of structs without packing
  • Using ++i vs i++ in loops (small but cumulative)
✅ Proven Optimisations
  • Cache storage variables into memory once per function
  • Pack multiple uint128 into a single storage slot
  • Use immutable and constant for values known at compile time
  • Short-circuit boolean evaluations (&& / ||)
  • Batch operations using multicall patterns

Deep Dive - Storage Variable Packing

One of the most impactful gas optimisation techniques is storage variable packing. The EVM stores data in 32-byte (256-bit) slots. If you declare two uint256 variables, they each occupy a full slot. But if you use smaller types like uint128, uint64, or uint32, multiple variables can fit in a single slot - saving 20,000 gas per slot avoided during writes (the cost of a cold SSTORE).

// UNOPTIMISED - uses 3 storage slots (60,000 gas for cold writes) contract BadPacking { uint256 public totalStaked; // slot 0 (full) uint256 public rewardRate; // slot 1 (full) uint256 public lastUpdateTime; // slot 2 (full) } // OPTIMISED - uses 2 storage slots (40,000 gas for cold writes) contract GoodPacking { uint256 public totalStaked; // slot 0 (full - needs 256 bits) uint128 public rewardRate; // slot 1 (first 128 bits) uint64 public lastUpdateTime; // slot 1 (next 64 bits) uint64 public rewardsDuration; // slot 1 (next 64 bits) - FREE! }

Notice that totalStaked stays as uint256 because staking amounts can be very large. But rewardRate and lastUpdateTime can safely be smaller types. The key is to place variables that can share a slot adjacent to each other in the declaration order. The Solidity compiler packs them automatically if they fit.

Deep Dive - The calldata vs memory Decision

When an external function receives array or struct parameters, you must choose between calldata and memory. The difference is significant:

  • calldata: Read-only, points directly to the transaction's input data. No copy is made. Costs ~3 gas per byte to access.
  • memory: Writable, but the EVM must copy the entire parameter from calldata into memory first. This copy costs ~3 gas per byte PLUS memory expansion costs.

The rule is simple: if you do not need to modify the parameter inside the function, always use calldata. For a function that receives a 100-element array, switching from memory to calldata can save 2,000-5,000 gas per call. Over millions of calls on a high-traffic MST dApp, this adds up to substantial savings.

08 · Practical Walkthrough

Auditing a Sample Staking Contract on MST

Let’s walk through a realistic example: a simple staking pool where users deposit MST tokens and earn rewards. The contract was written by a junior developer and contains several subtle issues. We’ll run Slither, Mythril, then manually fix the problems.

// SPDX-License-Identifier: MIT pragma solidity ^0.8.19; contract StakingPool { mapping(address => uint256) public stakes; mapping(address => uint256) public rewards; address public owner; constructor() { owner = msg.sender; } function stake() external payable { stakes[msg.sender] += msg.value; } function withdraw(uint256 amount) external { require(stakes[msg.sender] >= amount, "insufficient"); stakes[msg.sender] -= amount; (bool ok, ) = msg.sender.call{value: amount}(""); require(ok, "transfer failed"); } function claimReward() external { uint256 reward = rewards[msg.sender]; rewards[msg.sender] = 0; (bool ok, ) = msg.sender.call{value: reward}(""); require(ok); } function updateReward(address user, uint256 amount) external { require(msg.sender == owner, "only owner"); rewards[user] = amount; } }
⚠️ Issues Found by Slither

Slither immediately flags: reentrancy in withdraw() (state change after external call), unchecked return value in claimReward() (no require message), and missing events for all state changes. It also warns that owner could be centralised - a governance consideration.

💡 Fixes Applied

We reorder withdraw() to follow CEI: decrease stakes before the call. We add ReentrancyGuard to both withdraw and claimReward. We emit events StakeDeposited, Withdrawal, and RewardClaimed. We also change the updateReward function to use a role-based access control or timelock if it’s a governance parameter.

After applying fixes, we re-run Slither - zero Highs, zero Mediums. We then run Echidna with an invariant that total ETH balance of the contract must always equal sum of stakes + unclaimed rewards (minus any fees). Echidna finds no violations after 10,000 transactions. The contract is now ready for mainnet.

The Fixed Contract - Line by Line

Let's look at the corrected version of the staking contract after applying all audit findings. Every change is intentional and addresses a specific vulnerability.

// SPDX-License-Identifier: MIT pragma solidity ^0.8.19; import "@openzeppelin/contracts/security/ReentrancyGuard.sol"; import "@openzeppelin/contracts/access/Ownable.sol"; contract StakingPoolV2 is ReentrancyGuard, Ownable { mapping(address => uint256) public stakes; mapping(address => uint256) public rewards; event StakeDeposited(address indexed user, uint256 amount); event Withdrawal(address indexed user, uint256 amount); event RewardClaimed(address indexed user, uint256 amount); event RewardUpdated(address indexed user, uint256 newAmount); constructor() Ownable(msg.sender) {} function stake() external payable { require(msg.value > 0, "Must stake > 0"); stakes[msg.sender] += msg.value; emit StakeDeposited(msg.sender, msg.value); } function withdraw(uint256 amount) external nonReentrant { require(stakes[msg.sender] >= amount, "Insufficient stake"); // CEI: Effects before Interactions stakes[msg.sender] -= amount; (bool ok, ) = msg.sender.call{value: amount}(""); require(ok, "Transfer failed"); emit Withdrawal(msg.sender, amount); } function claimReward() external nonReentrant { uint256 reward = rewards[msg.sender]; require(reward > 0, "No rewards"); // CEI: Effects before Interactions rewards[msg.sender] = 0; (bool ok, ) = msg.sender.call{value: reward}(""); require(ok, "Reward transfer failed"); emit RewardClaimed(msg.sender, reward); } function updateReward(address user, uint256 amount) external onlyOwner { rewards[user] = amount; emit RewardUpdated(user, amount); } }

Key changes in the fixed version: (1) ReentrancyGuard imported and nonReentrant modifier applied to both withdrawal functions. (2) CEI pattern enforced - state changes happen before external calls. (3) Events emitted for every state change, enabling off-chain monitoring. (4) Input validation added (msg.value > 0). (5) Ownable from OpenZeppelin replaces manual owner tracking for battle-tested access control.

08B · Writing Your Audit Report

The Document That Opens Doors

An audit report is not just a list of bugs - it is a professional document that demonstrates your security competence. MST BuildGrant reviewers read audit reports carefully. A well-structured report can be the difference between a funded project and a rejected application. Your report should follow this structure:

1
Executive Summary
A half-page overview: what was audited, when, by whom, and the overall risk assessment (e.g., "No critical or high-severity issues remain after remediation").
2
Scope and Methodology
List every contract file, its purpose, and the tools used. Mention the Solidity compiler version, the chain (MST Mainnet, Chain ID 4646), and any external dependencies (OpenZeppelin version, oracle contracts, bridge contracts).
3
Findings Table
A table with columns: ID, Severity (Critical/High/Medium/Low/Info), Title, Description, Location (file:line), Status (Fixed/Acknowledged/Won't Fix), and Fix Description. Sort by severity, highest first.
4
Detailed Findings
For each finding, provide the vulnerable code snippet, an explanation of the attack scenario, a proof-of-concept (PoC) or test case, and the recommended fix. Include the fixed code snippet for comparison.
5
Gas Optimisation Appendix
Include the Hardhat Gas Reporter output before and after optimisation, with a summary of techniques applied and gas saved per function.

08C · Common Audit Pitfalls for Capstone Students

Mistakes That Cost Marks - And How to Avoid Them

❌ Pitfalls to Avoid
  • Running Slither once, seeing "0 issues," and calling it done - did you check the filter settings?
  • Ignoring informational findings - they often hint at deeper design issues
  • Copy-pasting the audit report template without customising it for your project
  • Auditing only the main contract but ignoring helper libraries or utility contracts
  • Not re-running tools after applying fixes - introducing new bugs during remediation is common
  • Writing "tested on testnet, works fine" as your entire security section
✅ Best Practices
  • Run Slither with --print human-summary for a high-level overview first
  • Write at least 3 Echidna invariants specific to your business logic
  • Have a peer review your audit report before submission - fresh eyes catch blind spots
  • Include a "Known Limitations" section that honestly discusses any accepted risks
  • Time-box your manual review: spend at least 30 minutes per 100 lines of Solidity
  • Test edge cases: zero amounts, max uint256, empty arrays, self-referencing addresses

Key Terms - Sub-Module 21.1

Slither
Static analysis tool for Solidity that detects vulnerabilities without executing code.
Mythril
Symbolic execution engine that explores all possible code paths to find deep bugs.
Echidna
Fuzzer for EVM bytecode; tests user-defined invariants with random transactions.
CEI
Checks-Effects-Interactions pattern - update state before external calls to prevent reentrancy.
Reentrancy Guard
A modifier that prevents a function from being re-entered within the same transaction.
Gas Optimisation
Techniques to reduce the computational cost of executing smart contract functions.
MST BuildGrant
MST ecosystem grant program (up to $10,000) that requires a security-audit report for eligibility.
FortunaX
MST’s provably fair random-number generation protocol, safe for gaming and lotteries.
Static Analysis
Examining source code without executing it to detect patterns that match known vulnerability signatures.
Symbolic Execution
A technique that explores all possible execution paths by treating inputs as symbolic variables rather than concrete values.
Invariant
A condition that must always hold true regardless of what transactions are executed (e.g., "total supply never exceeds max cap").
Fuzzing
Testing software by providing random, unexpected, or malformed inputs to trigger crashes or violations.
Storage Packing
Arranging Solidity state variables so that multiple small-type variables share a single 32-byte storage slot, reducing gas costs.
Front-Running
An attack where a malicious actor observes a pending transaction and submits their own transaction first with a higher gas price to profit.
Flash Loan Attack
An exploit that uses uncollateralised instant loans to manipulate prices or drain funds within a single transaction.
Delegatecall
A low-level EVM opcode that executes another contract's code in the caller's storage context - powerful but dangerous if misused in proxy patterns.
