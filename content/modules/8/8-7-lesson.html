02   Real-World Story

The Copied Code That Cost ₹10 Lakh

📖 Story

Karan was building a DeFi protocol with two types of staking pools. He wrote the first pool's smart contract from scratch, implementing ownership checks, pause functionality, and a withdrawal mechanism. It worked perfectly. For the second pool, he copied the entire 200-line contract, changed the staking logic slightly, and deployed it.


Three weeks after launch, a user discovered that Karan had forgotten to update the require statement in the second contract's withdrawal function. The first contract limited withdrawals to 10% per day. The copied contract had no limit. An attacker drained the entire pool. Over ₹10 lakh worth of tokens were gone. The bug was not in the new logic - it was in the duplicated boilerplate that Karan had failed to synchronize between the two contracts.


If Karan had used inheritance, he would have written the ownership and pause logic once, in a base contract, and both staking pools would have inherited it. A single fix to the base contract would have fixed both. This is the power of inheritance: write once, reuse everywhere, fix once, secure everywhere. Combined with interfaces, which enforce standards, inheritance is what makes the composable "money lego" ecosystem of blockchain possible.

03   The Core Concept

What Is Inheritance? The "Is-A" Relationship

Inheritance is a mechanism where one contract derives properties and behavior from another. The child contract inherits state variables and functions from the parent, and can override them or add new ones. This creates a hierarchical relationship that models real-world categories.

FIGURE 8.7-A · Visualizing Inheritance in Solidity
                          ┌─────────────────────────┐
                          │        Ownable          │  ← Base contract: ownership logic
                          │  - owner: address       │
                          │  + onlyOwner() modifier │
                          │  + transferOwnership()  │
                          └───────────┬─────────────┘
                                      │  "is"
                          ┌───────────┴─────────────┐
                          │        MyToken          │  ← Derived contract: inherits ownership
                          │  (is Ownable)           │
                          │  - balances: mapping    │
                          │  + mint()               │
                          │  + transfer()           │
                          └─────────────────────────┘

        MyToken now has: owner, onlyOwner, transferOwnership, balances, mint, transfer.
        It "is‑a" Ownable contract with extra token functionality.
Solidity inheritance is declared with the is keyword. The child contract merges the parent's state and logic.

In Solidity, you declare inheritance using the is keyword: contract Child is Parent { }. The child contract automatically gets access to all non-private state variables and functions of the parent. This is the foundation of code reuse in smart contracts.

🇮🇳 Indian Context Analogy

Think of inheritance like the organizational structure of the Indian Railways. At the top, the Railway Board defines common policies: safety regulations, ticketing standards, and employee rules. These are the "base contract." Each Zonal Railway (Northern, Southern, Western, etc.) inherits all these policies but can override specific ones - for example, Southern Railway might have different monsoon-preparedness procedures. Individual Divisions within each zone inherit from both the zone and the board, adding their own local rules.


If the Railway Board updates a safety regulation, every zone and division automatically follows the new rule - unless they have explicitly overridden it. This is exactly how Solidity inheritance works: a change in the base contract propagates to all child contracts that have not overridden that specific function. Without inheritance, every zone would maintain its own complete copy of all policies - and when a critical update happens, someone would inevitably forget to update one copy. That is the ₹10 lakh bug from our story.

There are three visibility levels that determine what a child contract can access from its parent:

🟢
public - Accessible from anywhere: other contracts, external calls, and child contracts. Automatically generates a getter function for state variables.
🟡
internal - Accessible only within the contract itself and its children. This is the default visibility for state variables. Think of it as "family-only access."
🔴
private - Accessible only within the contract that defines it. Children CANNOT access private members. The data still exists on-chain (nothing is truly hidden on a public blockchain), but the Solidity compiler will prevent child contracts from reading or writing private variables directly.
04   Basic Inheritance

The "is" Keyword and the Ownable Pattern

The most common inheritance pattern in blockchain development is the Ownable contract. OpenZeppelin's Ownable contract provides basic authorization: it stores the owner's address, provides a modifier that restricts access to the owner, and allows ownership transfer.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Ownable {
    address public owner;

    // Event emitted when ownership is transferred
    event OwnershipTransferred(address indexed previousOwner, address indexed newOwner);

    // The deployer becomes the initial owner
    constructor() {
        owner = msg.sender;
    }

    // Modifier: only the current owner can call the function
    modifier onlyOwner() {
        require(msg.sender == owner, "Not the owner");
        _;
    }

    // Transfer ownership to a new address
    function transferOwnership(address newOwner) public onlyOwner {
        require(newOwner != address(0), "New owner is zero address");
        emit OwnershipTransferred(owner, newOwner);
        owner = newOwner;
    }
}

// Now any contract can inherit this ownership logic
contract MySecureContract is Ownable {
    uint256 public secretValue;

    // Only the owner can set this value
    function setSecret(uint256 newValue) public onlyOwner {
        secretValue = newValue;
    }
}

In this example, MySecureContract inherits owner, onlyOwner, and transferOwnership from Ownable. It adds its own secretValue state variable and setSecret function. The onlyOwner modifier, defined in the parent, works perfectly in the child - because the child is an Ownable contract.

05   Overriding Functions

virtual and override - Changing Inherited Behavior

Sometimes a child contract needs to change how a parent's function works. Solidity provides the virtual and override keywords for this purpose. The parent marks a function as virtual to signal "I allow my children to override this." The child marks its version of the function with override to signal "I am intentionally replacing the parent's implementation."

contract Animal {
    function makeSound() public pure virtual returns (string memory) {
        return "...";
    }
}

contract Dog is Animal {
    // override replaces the parent's version
    function makeSound() public pure override returns (string memory) {
        return "Woof!";
    }
}

contract Cat is Animal {
    function makeSound() public pure override returns (string memory) {
        return "Meow!";
    }
}

Important Rules:

1. Only functions marked virtual can be overridden.

2. Overriding functions must use the override keyword.

3. The overriding function must have the same visibility or broader (e.g., external can be overridden as public, but not the reverse).

4. Mutability (view, pure, payable) must remain compatible - a non-payable function cannot be overridden as payable.

5. If overriding a function from multiple parents, you must explicitly list them: override(Parent1, Parent2).

06   Constructors

Constructors in Inheritance Chains

When a child contract is deployed, the constructors of all its parent contracts execute, from the most base-like to the most derived. If a parent constructor requires arguments, the child must supply them in its constructor definition.

contract Base {
    uint256 public creationTime;
    constructor(uint256 time) {
        creationTime = time;
    }
}

contract Derived is Base(42) {  // Passes 42 to Base's constructor
    // Derived gets creationTime = 42
}

contract DerivedWithParam is Base {
    constructor(uint256 time) Base(time) {  // Passes deployment arg to Base
        // Base's constructor runs before Derived's body
    }
}
🔑 Constructor Execution Order

SolC (the Solidity compiler) enforces that base contracts are initialized before derived contracts. This ensures that state variables set in parent constructors are available when the child constructor executes. In multiple inheritance, the order follows C3 linearization (explained below).

07   Multi-Level Inheritance

Grandparent → Parent → Child

A contract can inherit from another contract that itself inherits from a third. This creates a chain: Grandparent → Parent → Child. The child inherits everything from the parent, which includes everything from the grandparent.

contract Grandparent {
    function generation() public pure returns (string memory) {
        return "I am the root";
    }
}

contract Parent is Grandparent {
    function generation() public pure virtual override returns (string memory) {
        return "I am the middle";
    }
}

contract Child is Parent {
    function generation() public pure override returns (string memory) {
        return "I am the leaf";
    }
}

Deploying the Child contract creates a single contract on the blockchain that contains all three generations' code. Calling generation() returns "I am the leaf" - the most derived override wins.

08   Multiple Inheritance

The Diamond Problem and C3 Linearization

Solidity allows a contract to inherit from multiple parent contracts. This creates a potential conflict: if two parents define the same function, which version does the child use? This is the classic Diamond Problem. Solidity solves it using C3 Linearization, an algorithm that creates a single, unambiguous order of all ancestor contracts.

FIGURE 8.7-B · C3 Linearization Solving the Diamond Problem
                    ┌──────────┐
                    │    A     │  ← Base (virtual)
                    │  foo()   │
                    └────┬─────┘
                         │
              ┌──────────┴──────────┐
              │                     │
        ┌─────┴─────┐         ┌────┴──────┐
        │    B      │         │     C     │  ← Both override A
        │ override  │         │ override  │
        └─────┬─────┘         └────┬──────┘
              │                    │
              └──────────┬─────────┘
                         │
                   ┌─────┴─────┐
                   │     D     │  ← inherits B, C
                   │ override(B, C) │
                   └───────────┘

        C3 Linearization of D: [D, B, C, A]
        When D calls foo(): D's override executes.
        D's foo() can call super.foo() which resolves to B.foo(),
        which can call super.foo() which resolves to C.foo(),
        which can call super.foo() which resolves to A.foo().
C3 linearization creates a single inheritance path, resolving the ambiguity of which parent's function to call.
contract A {
    function foo() public pure virtual returns (string memory) {
        return "A";
    }
}

contract B is A {
    function foo() public pure virtual override returns (string memory) {
        return string.concat("B", super.foo()); // "BA"
    }
}

contract C is A {
    function foo() public pure virtual override returns (string memory) {
        return string.concat("C", super.foo()); // "CA"
    }
}

contract D is B, C {
    function foo() public pure override(B, C) returns (string memory) {
        return string.concat("D", super.foo()); // "DBCA"
    }
}

In this example, calling D.foo() returns "DBCA". The super keyword follows C3 linearization: D → B → C → A. This is critical for understanding how initializers, constructors, and function calls flow in complex inheritance structures.

🔬 Click to expand: The C3 Linearization Algorithm Explained

C3 linearization is the algorithm Python and Solidity use to resolve method resolution order. The rule is: a child contract always comes before its parents, and the order of parents as written in the contract declaration is preserved.

For contract D is B, C { }, the linearization is [D, B, C, A]. This satisfies: D before B and C, B before A (since B is A), C before A (since C is A), B before C (because in D's declaration, B comes first). If the order had been contract D is C, B { }, the linearization would be [D, C, B, A].

08-B   Decision Guide

When to Use What - Inheritance vs. Interfaces vs. Abstract Contracts vs. Libraries

Solidity gives you four primary tools for code organization and reuse. Choosing the wrong one leads to bloated bytecode, wasted gas, or inflexible designs. Here is a practical decision framework:

09   Abstract Contracts

The Blueprint - abstract Contracts

An abstract contract is a contract that cannot be deployed directly. It exists only to be inherited by other contracts. It may contain unimplemented functions (functions without a body) that child contracts must implement. This is how Solidity defines templates and standards.

abstract contract Shape {
    // Implemented function – child can inherit as‑is
    function description() public pure returns (string memory) {
        return "I am a shape";
    }

    // Unimplemented function – child MUST implement this
    function area() public view virtual returns (uint256);
}

contract Circle is Shape {
    uint256 public radius;

    constructor(uint256 r) {
        radius = r;
    }

    function area() public view override returns (uint256) {
        return 3 * radius * radius; // Simplified!
    }
}

Any contract that inherits from an abstract contract and does not implement all its unimplemented functions becomes abstract itself. Abstract contracts are ideal for creating base templates where some behavior is shared but specific implementations must be provided by the derived contracts.

10   Interfaces

The Contract Standard - Interfaces

An interface is the strictest form of an abstract contract. It defines what functions a contract must have, but not how they work. Interfaces are the backbone of interoperability in the EVM ecosystem - they are what allow wallets to display your token balance, exchanges to list your token, and other contracts to interact with your contract without knowing its internal code.

🔑 Rules of Interfaces

1. No state variables. None. Period.

2. No constructors. An interface cannot be deployed.

3. No function bodies. All functions must be external (or public, if defined as interface).

4. No modifiers with implementation (only function signatures).

5. Cannot inherit from other contracts, but can inherit from other interfaces.

The most famous interface in blockchain is IERC20. Any contract that implements IERC20 is recognized as a fungible token by wallets, exchanges, and dApps.

// The IERC20 interface (simplified)
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}

// Our custom token promises to follow the IERC20 standard
contract MyToken is IERC20 {
    mapping(address => uint256) private _balances;
    // ... implementation of all IERC20 functions ...
}

By declaring contract MyToken is IERC20, we promise that MyToken has all the functions IERC20 requires. If we forget to implement totalSupply(), the compiler will refuse to compile. This is a powerful enforcement mechanism that prevents catastrophic integration errors.

10-B   Deep Dive

IERC-721 - How the NFT Standard Uses Interfaces and Inheritance

The ERC-721 standard defines Non-Fungible Tokens (NFTs). Unlike ERC-20 tokens where every unit is identical (fungible), each ERC-721 token has a unique ID. This is the standard behind every digital art NFT, in-game asset, and digital land deed on Ethereum, Polygon, and other EVM chains. Understanding how ERC-721 is structured teaches you how interfaces and inheritance work together in a real production system.

The IERC-721 interface defines the minimum set of functions any NFT contract must implement:

// Simplified IERC-721 Interface
interface IERC721 {
    event Transfer(address indexed from, address indexed to, uint256 indexed tokenId);
    event Approval(address indexed owner, address indexed approved, uint256 indexed tokenId);
    event ApprovalForAll(address indexed owner, address indexed operator, bool approved);

    function balanceOf(address owner) external view returns (uint256 balance);
    function ownerOf(uint256 tokenId) external view returns (address owner);
    function safeTransferFrom(address from, address to, uint256 tokenId) external;
    function transferFrom(address from, address to, uint256 tokenId) external;
    function approve(address to, uint256 tokenId) external;
    function setApprovalForAll(address operator, bool approved) external;
    function getApproved(uint256 tokenId) external view returns (address operator);
    function isApprovedForAll(address owner, address operator) external view returns (bool);
}

When you build an NFT project - say, a digital art marketplace for Indian artists - you do not implement IERC721 from scratch. You inherit from OpenZeppelin's ERC721 contract, which already implements the interface. Then you layer on additional functionality through multiple inheritance:

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/token/ERC721/extensions/ERC721URIStorage.sol";
import "@openzeppelin/contracts/token/ERC721/extensions/ERC721Burnable.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

// Multiple inheritance: this contract IS an ERC721, IS URI-storable,
// IS burnable, and IS ownable - all at once.
contract IndianArtNFT is ERC721, ERC721URIStorage, ERC721Burnable, Ownable {
    uint256 private _nextTokenId;

    constructor() ERC721("IndianArtNFT", "IANFT") Ownable() {}

    function mintArt(address artist, string memory metadataURI)
        public onlyOwner returns (uint256)
    {
        uint256 tokenId = _nextTokenId++;
        _safeMint(artist, tokenId);
        _setTokenURI(tokenId, metadataURI);
        return tokenId;
    }

    // Required overrides when inheriting from multiple parents
    // that define the same function
    function tokenURI(uint256 tokenId)
        public view override(ERC721, ERC721URIStorage)
        returns (string memory)
    {
        return super.tokenURI(tokenId);
    }

    function supportsInterface(bytes4 interfaceId)
        public view override(ERC721, ERC721URIStorage)
        returns (bool)
    {
        return super.supportsInterface(interfaceId);
    }
}
🔑 Why Multiple Overrides Are Required

Notice the override(ERC721, ERC721URIStorage) syntax. Both ERC721 and ERC721URIStorage define tokenURI() and supportsInterface(). When a child inherits from both, the compiler does not know which parent's version to use. You must explicitly override and resolve the conflict. Using super.tokenURI(tokenId) calls the function according to C3 linearization order, ensuring all parents in the chain get their say.

🇮🇳 Real-World Application

Consider WazirX's NFT marketplace or a startup building on Polygon for the Indian creator economy. Indian digital artists upload their work, and the platform mints an ERC-721 token representing ownership. The ERC721URIStorage extension stores the IPFS link to the artwork's metadata. ERC721Burnable allows artists to permanently destroy a token if they want to remove a piece from circulation. Ownable restricts minting to the platform's admin wallet. All of this is achieved through four lines of inheritance - no need to write thousands of lines of token logic from scratch.

The supportsInterface() function deserves special attention. It implements ERC-165, a standard that allows contracts to declare which interfaces they support. When a marketplace like OpenSea queries your contract, it calls supportsInterface(0x80ac58cd) (the interface ID for ERC-721). If your contract returns true, the marketplace knows it can safely call ownerOf(), transferFrom(), and other ERC-721 functions. This is interface-driven interoperability in action.

11   Libraries

using for - Gas-Efficient Code Reuse with Libraries

A library is a special type of contract that cannot hold state or receive ether. It is deployed once at a specific address, and other contracts call its functions using DELEGATECALL, meaning the library's code executes in the context of the calling contract's storage. Libraries are ideal for reusable utility functions like math operations, array manipulations, and string processing.

library MathLib {
    function max(uint256 a, uint256 b) internal pure returns (uint256) {
        return a > b ? a : b;
    }

    function min(uint256 a, uint256 b) internal pure returns (uint256) {
        return a < b ? a : b;
    }
}

contract Calculator {
    using MathLib for uint256;

    function findMax(uint256 x, uint256 y) public pure returns (uint256) {
        return x.max(y); // Calls MathLib.max(x, y)
    }
}

The using for directive attaches library functions to a specific type. In the example, using MathLib for uint256 makes max and min available as member functions on any uint256 variable. This is purely syntactic sugar - the compiler transforms it into a regular library call. Libraries keep your contracts modular, reusable, and gas-optimized.

11-B   OpenZeppelin Libraries

Common OpenZeppelin Libraries Every Developer Should Know

OpenZeppelin provides a collection of battle-tested libraries that handle common operations. Understanding their history and purpose will save you from reinventing the wheel - and from introducing vulnerabilities that have already been solved by the community.

01
SafeMath (Historical - No Longer Needed)

Before Solidity 0.8.0, arithmetic operations silently overflowed. Adding 1 to the maximum uint256 value would wrap around to 0 - a catastrophic bug that was exploited in multiple real attacks, including the 2018 BEC token hack. OpenZeppelin's SafeMath library provided add(), sub(), mul(), and div() functions that reverted on overflow. Since Solidity 0.8.0, the compiler checks for overflow by default, making SafeMath unnecessary. However, you will still see it in legacy codebases and older tutorials. If you encounter using SafeMath for uint256 in a contract running Solidity 0.8+, it is redundant and wastes gas.

02
Address Library

The Address library provides utility functions for working with Ethereum addresses. Its most important function is sendValue(address payable recipient, uint256 amount), which safely sends Ether and reverts if the transfer fails. It also provides functionCall() for low-level calls with automatic revert message propagation, and isContract() to check if an address contains deployed code.

03
Strings Library

Solidity has limited built-in string support. The Strings library provides toString(uint256) to convert numbers to their string representation, toHexString(uint256) for hexadecimal conversion, and equal(string, string) for string comparison. These are essential when generating on-chain metadata for NFTs or building human-readable error messages.

04
Counters (Deprecated in v5.x)

The Counters library provided a simple incrementing counter, commonly used for generating sequential token IDs in NFT contracts. It wrapped a uint256 in a struct and provided increment(), decrement(), and current() functions. In OpenZeppelin v5.x, this library was removed in favor of simply using a plain uint256 variable with direct increment (++), as the added abstraction provided no real safety benefit.

// Using OpenZeppelin's Strings library for on-chain NFT metadata
import "@openzeppelin/contracts/utils/Strings.sol";

contract OnChainNFT {
    using Strings for uint256;

    function generateName(uint256 tokenId) public pure returns (string memory) {
        // Converts tokenId (a number) to a string for metadata
        return string.concat("Indian Art #", tokenId.toString());
    }

    function generateHexId(uint256 tokenId) public pure returns (string memory) {
        return tokenId.toHexString();
    }
}
11-C   Gas Implications

How Inheritance Affects Gas Costs and Storage Layout

Inheritance is a compile-time mechanism. When you deploy a child contract, the EVM does not maintain separate "parent" and "child" sections. The Solidity compiler flattens the entire inheritance tree into a single contract bytecode. This has important implications for gas costs and contract size.

⛽ Storage Layout Across Parents

State variables from parent contracts occupy storage slots in the order of C3 linearization. If contract A declares uint256 x (slot 0) and contract B declares uint256 y (slot 1), then contract C is A, B will have x at slot 0 and y at slot 1. This ordering is critical when working with upgradeable contracts (proxy patterns) - if you change the inheritance order or insert a new parent, storage slots shift, and all existing data becomes corrupted.

Key gas considerations:

⛽
Contract deployment cost increases with inheritance depth. Every parent's code is copied into the child's bytecode. A contract that inherits from ERC721, ERC721URIStorage, ERC721Burnable, Ownable, and Pausable will have significantly larger bytecode than a minimal contract. Ethereum has a 24,576-byte contract size limit (EIP-170) - deep inheritance chains can push you toward this limit.
⛽
Function dispatch adds minimal overhead. The Solidity compiler generates a function selector lookup table at the beginning of the bytecode. More inherited functions mean a larger lookup table, but the actual runtime cost difference is negligible (a few gas units per additional function selector comparison).
⛽
super calls in deep chains cost more. When super.foo() traverses multiple parents in the C3 linearization, each function call adds to the call stack. In a chain of 5 contracts, calling super.foo() from the most derived contract executes 5 function bodies sequentially. This is usually acceptable but can matter in gas-sensitive DeFi operations on networks like Polygon where every wei counts for high-frequency transactions.
⛽
Libraries with internal functions are inlined. When a library function is declared internal, the compiler copies its bytecode directly into the calling contract - no DELEGATECALL overhead. External library functions use DELEGATECALL, which costs approximately 2,600 gas for the call itself plus the gas for execution. Choose internal library functions for frequently called utilities and external for rarely used, large functions to save on deployment size.
11-D   Common Pitfalls

Inheritance Bugs That Catch Even Experienced Developers

Inheritance is powerful but introduces subtle bugs that are difficult to detect in testing and can lead to critical vulnerabilities in production. Here are the most common pitfalls and how to avoid them.

⚠️ Pitfall 1: Shadowing State Variables

If a child contract declares a state variable with the same name as one in the parent, the child creates a completely separate variable. The parent's version still exists in storage but the child's functions will reference the new one. This is called shadowing and leads to extremely confusing bugs where the parent's functions read one value and the child's functions read a different value.

contract Parent {
    uint256 public value = 100;

    function getValue() public view returns (uint256) {
        return value;  // Returns Parent's 'value' (100)
    }
}

contract Child is Parent {
    uint256 public value = 200;  // SHADOWING! Creates a NEW variable

    function getChildValue() public view returns (uint256) {
        return value;  // Returns Child's 'value' (200)
    }
    // But calling getValue() still returns 100!
    // The parent's function references the parent's 'value'.
}
// Solidity 0.6+ emits a compiler WARNING for shadowed variables.
// Always heed compiler warnings - they prevent real losses.
⚠️ Pitfall 2: Constructor Ordering Bugs

When a contract inherits from multiple parents, constructor execution order follows C3 linearization - NOT the order you pass arguments. This means the constructor you think runs first might actually run second. If one parent's constructor depends on state set by another parent's constructor, the wrong order causes initialization bugs.

contract A {
    uint256 public x;
    constructor(uint256 _x) { x = _x; }
}

contract B {
    uint256 public y;
    constructor(uint256 _y) { y = _y; }
}

// The order in the "is" clause determines constructor order!
contract C is A, B {
    // A's constructor runs first, then B's
    constructor() A(10) B(20) {}
}

contract D is B, A {
    // B's constructor runs first, then A's
    constructor() A(10) B(20) {}
    // Note: even though A(10) appears before B(20) in the
    // constructor arguments, B runs first because D "is B, A".
}
⚠️ Pitfall 3: Forgetting the override Keyword

If you intend to override a parent's function but forget the override keyword, the compiler will throw an error (in Solidity 0.6+). However, a more subtle issue occurs when you accidentally create a function with the same name but a different signature (different parameters). This creates function overloading, not overriding - the parent's function still exists alongside your new one, leading to unexpected behavior when external callers invoke the parent's version.

⚠️ Pitfall 4: The "Unreachable Parent" Problem

In multi-level inheritance, if you override a function in a middle contract but forget to mark it as virtual, the grandchild contract cannot override it further. This locks behavior at the middle layer and prevents further customization. Always mark overriding functions as virtual override if there is any possibility that a deeper child might need to customize them.

contract Base {
    function greet() public pure virtual returns (string memory) {
        return "Hello from Base";
    }
}

contract Middle is Base {
    // BUG: Missing 'virtual' keyword!
    function greet() public pure override returns (string memory) {
        return "Hello from Middle";
    }
}

// This will NOT compile:
// contract Child is Middle {
//     function greet() public pure override returns (string memory) {
//         return "Hello from Child";
//         // ERROR: Middle.greet() is not virtual
//     }
// }

// FIX: Middle should use "virtual override" instead of just "override"
12   Hands-On Integration

Using OpenZeppelin Contracts - The Standard Library

OpenZeppelin provides battle-tested, community-audited base contracts for all common blockchain functionality. You should never write your own token standard, access control, or math library from scratch for production. Use OpenZeppelin and extend it through inheritance.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract MyGovernanceToken is ERC20, Ownable {
    // ERC20 provides: transfer, approve, transferFrom, balanceOf, totalSupply...
    // Ownable provides: owner, onlyOwner, transferOwnership...

    constructor() ERC20("MyGovernanceToken", "MGT") Ownable() {
        // Mint 1 million tokens to the deployer
        _mint(msg.sender, 1000000 * 10 ** decimals());
    }

    // Only the owner can mint additional tokens
    function mint(address to, uint256 amount) public onlyOwner {
        _mint(to, amount);
    }
}

With just these few lines, we have created a fully functional ERC-20 token with ownership management, using code that has been audited by the top security firms in the world. This is the power of inheritance combined with interfaces - you build on the shoulders of giants.

14   Glossary

Key Terms - Sub-Module 8.7

Inheritance
A mechanism where a contract derives properties and behavior from another contract, using the is keyword.
virtual / override
Keywords used for polymorphism. Parent functions are marked virtual; child functions that replace them use override.
C3 Linearization
The algorithm Solidity uses to resolve multiple inheritance, creating a single unambiguous order for function resolution.
Abstract Contract
A contract that cannot be deployed and may contain unimplemented functions. Used as a base template.
Interface
A strict abstract contract defining external function signatures with no state, constructors, or function bodies.
Library
A stateless contract deployed once and called via DELEGATECALL, used for reusable utility functions.
using for
A directive that attaches library functions to a specific type, enabling cleaner syntax like myArray.push(5).
super
A keyword that calls the next function in the C3 linearization order, not necessarily the immediate parent.
DELEGATECALL
A low-level EVM opcode that executes another contract's code in the context of the calling contract's storage. Libraries use DELEGATECALL so their logic can read and write the caller's state variables.
Diamond Problem
An ambiguity that arises in multiple inheritance when two parent contracts define the same function. Solidity resolves this using C3 linearization, creating a single unambiguous call order.
EIP-2535 (Diamond Pattern)
An Ethereum Improvement Proposal that defines a proxy pattern allowing a single contract address to delegate calls to multiple implementation contracts (called "facets"). Used to overcome the 24 KB contract size limit while maintaining a unified interface.
Polymorphism
The ability of different contracts to respond to the same function call in different ways. In Solidity, polymorphism is achieved through the virtual and override keywords - a child contract can replace a parent's function with its own implementation.
Shadowing
A bug where a child contract declares a state variable with the same name as the parent, creating two separate variables instead of one shared variable. The Solidity compiler issues a warning when this occurs.
ERC-165
A standard that allows contracts to declare which interfaces they support via the supportsInterface() function. Marketplaces and wallets use ERC-165 to discover a contract's capabilities before interacting with it.
